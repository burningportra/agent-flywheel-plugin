# Security review — 2026-04-23

Reviewer: fresh-eyes security agent (Claude Opus 4.7), read-only pass
Scope: commit range `dfc8c51..HEAD` (19 commits, +18672/-512 LOC)
Beads under audit: mq3 (path-safety), 016 (clone-safety), 8tf (fs-safety), plus telemetry P1-2/3/4 claims (commit `2f94b5b`).

## Summary

**Yellow.** The three P0 defenses (mq3 path-safety, 016 clone-safety, 8tf fs-safety) are fundamentally sound — the primitives are correctly built, reject the obvious attack inputs (`..`, null bytes, control chars, non-allowlisted hosts, user-owned paths) and pass their unit tests. However, the defenses are not applied everywhere they need to be. The most serious gap is in `mcp-server/src/tools/emit-codex.ts` where the companion argument `pluginRoot` is accepted *unvalidated* — defeating mq3 at exactly the boundary mq3 was built to protect. There is also a category-wide symlink-follow gap: `assertSafeRelativePath` explicitly defers realpath resolution to callers, and no caller does it, so a pre-placed symlink inside the project root still routes writes anywhere on disk. P1-2/P1-3/P1-4 fixes in `telemetry.ts` and `agent-mail.ts` check out as real and correct. Ship only after closing P0-1 below; the rest can wait a cycle.

## P0 findings (must-fix before ship)

### P0-1. `flywheel_emit_codex` accepts unvalidated `pluginRoot`

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/tools/emit-codex.ts:98`
- Proof:
  ```
  const pluginRoot = args.pluginRoot ?? cwd;   // ← never sanitized
  ...
  const report = await emitCodex({ pluginRoot, targetDir: absoluteTarget });
  ```
  The handler sanitizes `targetDir` via `assertSafeRelativePath`, but `pluginRoot` passes straight through. In `emit/codex.ts:513` that value is joined as `<pluginRoot>/skills` and every matching `SKILL.md` on disk is read. Attacker payload: `{ targetDir: ".codex-out", pluginRoot: "/" }` → server walks the root filesystem looking for `skills/<name>/SKILL.md`, reads anything it finds, and writes the bodies verbatim into `<cwd>/.codex-out/.codex/skills/*.md` and `AGENTS.md`. On a developer machine with `~/skills/foo/SKILL.md` layouts (common with Claude Code), that leaks arbitrary file contents into a project directory the attacker can then `cat`.
- Impact: arbitrary file-read + write-into-project exfiltration. Low complexity for any caller that can set MCP tool args.
- Fix: run `assertSafeRelativePath(args.pluginRoot, { root: cwd, allowAbsoluteInsideRoot: true })` OR require `pluginRoot` to be a realpath under `cwd`. Tighter option: drop the `pluginRoot` arg entirely and always derive it from `ctx.cwd` (the comment already notes it is only there for the plugin checkout case — fold that into `ctx` instead).

## P1 findings (should-fix this cycle)

### P1-1. No realpath / symlink-follow check at filesystem boundaries

- Files:
  - `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/path-safety.ts:~230` (comment: "we rely on realpath resolution being performed separately where needed")
  - `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/fs-safety.ts:~95` (`isFlywheelManagedPath` docstring: "Callers that want symlink-escape protection should pre-resolve with `fs.realpathSync`")
  - `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/tools/emit-codex.ts:120`
- Proof: neither `assertSafeRelativePath` nor `isFlywheelManagedPath` calls `fs.realpathSync`. The comment is explicit that the caller is responsible — but `grep -rn realpath mcp-server/src/` shows no caller actually does. Exploit sketch: a malicious `bead-pack/install.sh` (or any prior-tool-call-controllable write) creates a symlink `<cwd>/safe-dir -> /etc`. A later `flywheel_emit_codex({ targetDir: "safe-dir" })` passes path-safety (`relative(cwd, resolve(cwd, "safe-dir")) === "safe-dir"`), but `emit/codex.ts` then `fs.mkdir("<cwd>/safe-dir/.codex/skills", { recursive: true })` follows the symlink and writes outside the project.
- Fix: in the MCP handler, after `safe.ok`, call `realpathSync(absoluteTarget)` (or `realpath` async) and re-run the `relative(root, realpath).startsWith("..")` containment check. Same for every `guardedUnlink`/`guardedRemoveDir` call site. Alternatively tighten the library — add an option `{ requireRealpathInside: true }` that does the check centrally. This pattern is the exact CE phase-4 blunder #3 described in `fs-safety.ts` — worth re-reading the doc and plugging the remaining hole.

### P1-2. `backupThenReplace` grows `.pi-flywheel/_backup/` unbounded

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/fs-safety.ts:161`
- Proof: every call appends a `<ISO-with-nanos>/<basename>` subtree; nothing deletes old entries. A long-running session (telemetry flush, spec regen, checkpoint rotation, etc.) accrues backups forever. Not a security issue by itself, but reaches into security territory when the user has sensitive files in `.pi-flywheel` — the backup keeps *every* historical version of error-counts.json etc. on disk indefinitely.
- Fix: add a retention pass (keep last N directories OR anything newer than M days) invoked from the same handler that calls `backupThenReplace`. Cheap to bolt on.

### P1-3. `isFlywheelManagedPath` allowlist hard-codes `mcp-server/dist`

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/fs-safety.ts:~45` (`FLYWHEEL_MANAGED_DIRS`)
- Proof: `FLYWHEEL_MANAGED_DIRS = [".pi-flywheel", ".pi-flywheel-feedback", join("mcp-server", "dist")]`. When the plugin is installed as a dependency (not the plugin checkout), `mcp-server/dist` under the consumer's `cwd` is *not* flywheel-managed — it's the consumer's own file tree. `guardedUnlink` will happily delete or overwrite user files there if any tool targets them.
- Fix: remove `mcp-server/dist` from the allowlist, or scope it to a sentinel detected by presence of `package.json` `name === "agent-flywheel-plugin"`.

## P2 findings (nice-to-have)

### P2-1. Telemetry error messages can echo stdin / paths

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/telemetry.ts` (log.warn calls in `storeTelemetry`, `withFlushLock`)
- Proof: `log.warn('telemetry_store_failed: unexpected error during flush', { err: err instanceof Error ? err.message : String(err) })`. An `err.message` from Node fs can contain absolute paths that leak the user's home directory. Not a secret per se, but post-mortem exports may end up in remote logs / shared artifacts. The `recentEvents` ring buffer only carries `{code, ts, ctxHash?}` — no message/stack — so the aggregator itself is clean. The leak is only in the *logger output*.
- Fix: pass errors through a small redactor (`${os.homedir()}` → `~`) before `log.warn`, or route the logger through the existing `normalizeText` helper.

### P2-2. `validateCloneUrl` only rejects `--`-prefix args *after* URL parsing

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/clone-safety.ts:~40`
- Proof: the order is `rawUrl.startsWith('-')` then `new URL(...)` then allowlist. Good. Observation only — when `extraAllowedHosts` is passed by a caller that took the host from user input (e.g. a config file), a TOCTOU is possible between config read and `validateCloneUrl`. Low-probability path; documented as a caveat so a future caller doesn't accidentally wire user input straight into `extraAllowedHosts`.
- Fix: add a JSDoc warning on `extraAllowedHosts` that it MUST come from a source the user controls via policy, not per-call input.

### P2-3. Insecure-protocol escape hatch `FLYWHEEL_ALLOW_INSECURE_CLONE=1`

- File: `/Users/kevtrinh/Documents/GitHub/agent-flywheel-plugin/mcp-server/src/utils/clone-safety.ts:~90`
- Proof: env-var bypass is sensible, but there's no audit trail — a subprocess that inherits the env var silently gets the insecure clone path. No log warning is emitted.
- Fix: `log.warn({ code: 'insecure_clone_enabled' }, ...)` once per process on first use.

## Verified strengths

- **`assertSafeRelativePath` rejects the four canonical traversal inputs correctly.** Walked the function against `"/etc/passwd"` (caught by `isAbsolute && !allowAbsoluteInsideRoot`), `"../../../etc/passwd"` (caught by `normalize(input).split(sep).some(s => s === "..")`), `"foo/../../../etc/passwd"` (same check on normalized form — `normalize` keeps the leading `..` after cancellation), `"foo\0bar"` (null-byte check), and control chars (`CONTROL_CHAR_RE`). The two-layer check (static rejection + `relative(root, resolve(root, input))` containment) is defense-in-depth.
- **`assertSafeSegment` gets the CE phase-4 colon lesson right.** Explicitly rejects `:` with a comment tying it back to the opencode blunder. Reserved-segment set includes `.`, `..`, `.git` — appropriate.
- **`exec.ts` uses `spawn` with `shell: false`.** `safeClone` passes `--` before the URL (belt + suspenders if the URL starts with `-`), does not `--recurse-submodules`, uses `--single-branch --no-tags`, and pins HEAD SHA after clone with a regex validator (`^[0-9a-f]{7,64}$`). This is genuinely well-built.
- **`validateCloneUrl` strips credential URLs.** `parsed.username || parsed.password → invalid_url`. Also rejects embedded control chars, `-`-prefix, non-https without explicit env opt-in, and non-allowlisted hosts. The `github.com.evil.com` trick fails because `parsed.hostname` is the full host and the allowlist uses `Set.has` (exact match).
- **P1-3 `error-counts.lock` O_EXCL is real.** `await open(lockPath, 'wx')` with `FLUSH_LOCK_MAX_ATTEMPTS` retry and `finally { unlink(lockPath) }` is the correct pattern. Lock path itself is inside `.pi-flywheel` and gated by `isFlywheelManagedPath` before acquire (`telemetry.ts:~310`). Verified.
- **P1-4 `bootstrapCoordinator` in-process dedupe is real.** `_bootstrapInFlight = new Map<string, Promise<...>>`, key includes `cwd`, `agentName`, `program` separated by `\u0000`, slot cleared in `finally` only if the current occupant is still this promise. Correct pattern. Verified by `agent-mail.test.ts:243+`.
- **P1-2 global ring-buffer cap is real.** `buildSnapshot` takes top-N codes then slices `recentEvents` to `maxEvents`; the in-memory cap is enforced per-record in `recordErrorCode` (evicts from largest bucket). Worst-case footprint bounded.
- **Telemetry `recentEvents` payload is minimal.** Only `{code, ts, ctxHash?}` — no freeform message, no stack, no env var, no path. `ctxHash` is the hash not the content. Good hygiene.
