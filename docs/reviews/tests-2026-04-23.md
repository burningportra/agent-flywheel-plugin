## Weak tests (count-bumps, mock-only, stale expectations)

1. **`errors.schema.test.ts` (19 its)** — The header comment claims "16
   legacy + 10 v3.4.0 additions" but the V3_4_CODES list now has **13
   entries** (including `wave_collision_detected`, `review_mode_gate_failed`,
   `review_headless_findings`) and asserts `toHaveLength(29)`. The
   comment/doc drift makes future reviewers think the file was count-bumped
   casually. Fix: update the JSDoc to say "16 legacy + 13 new".

2. **`error-contract.test.ts` (25 its)** — Verifies hints for
   `missing_prerequisite`, `concurrent_write`, `exec_timeout`,
   `blocked_state`, plus absence-of-hint. **It does NOT iterate all 29
   codes** to enforce "every FlywheelError has a hint." So new codes added
   without a hint (e.g. the four `wave_collision_detected` /
   `review_*` / `hotspot_*` / `template_*` / `telemetry_store_failed`) are
   **not regression-protected** by this suite. This is the #1 hint-contract
   regression risk.

3. **`coordination.collisions.test.ts` (29 its)** — All end-to-end paths
   use a mocked `exec` closure (not `child_process.spawn` against a real
   git repo). The glob matcher tests and `forceSerialRerun` tests are
   solid, but `detectWaveCollisions` never exercises a real
   multi-worktree scenario. Fix: add one integration test that sets up
   two real `git worktree` clones in a tmpdir.

4. **`refresh-learnings.test.ts` (24 its)** — Good Jaccard
   symmetry/identity/orthogonality tests, but I do **not** see explicit
   boundary tests at the documented thresholds **0.75** and **0.85** (dedup
   vs merge). Tests for "score exactly 0.749" and "score exactly 0.851"
   would catch off-by-one `>=` vs `>` bugs.

5. **`tools/review.test.ts` (36 its)** — Review-mode matrix has a case
   per mode (interactive, report-only, headless, autofix-dirty,
   autofix-clean) — this is the golden path the prompt requested. BUT the
   autofix flow is only tested via **string-in-preamble assertions**
   (`expect(task.task).toContain('Review mode: autofix')`). It does not
   verify the effect: that autofix actually applies patches. A pure-string
   check passes even if the mode flag is dead-wired.

6. **`codex-handoff.test.ts` (17 its)** — Excellent content: round-trip,
   clamp, hint-verbatim, stall simulation. Minor gap: `MAX_TEXT_FIELD_LENGTH
   = 2000` truncation is tested via `.repeat(LEN*3)` — no test for the
   **exact boundary** (length === 2000 vs 2001).

7. **`utils/path-safety.test.ts` (25 its)** — Strong coverage: traversal,
   null byte, control char, absolute-when-relative, segment separator,
   backslash, colon, reserved segment, maxLength, leading-dot gate. Two
   unchecked vectors: **Unicode normalization** (e.g. NFC vs NFD
   homograph) and **URL-encoded traversal** (`%2e%2e`). The implementation
   may implicitly handle these; tests should pin that behavior.

8. **`utils/fs-safety.test.ts` (21 its)** — Tests `backupThenReplace`
   idempotency and the "unique backup dir under rapid calls" case. The
   comment says the CE-phase4 blunder was ISO timestamps colliding — but
   the test only proves *two* rapid calls produce distinct dirs. It does
   NOT test true parallelism (e.g. 10 concurrent calls from
   `Promise.all`). Race conditions require real concurrency.

9. **`utils/text-normalize.test.ts` (10 its)** — Covers LF, CRLF, BOM+LF,
   BOM+CRLF, bare CR, empty string, lone BOM, interior BOM preservation,
   and an on-disk BOM+CRLF fixture. **Very thorough.** Only weakness:
   no test for *mixed* line endings within one string (e.g. `"a\nb\r\nc\rd"`).

10. **`adapters/model-diversity.test.ts` (19 its)** — Appears comprehensive
    for model-routing. Since this is also the **only** indirect consumer
    of `agent-names`, any agent-names bug must be caught here or nowhere.

## Pre-existing failure triage

**Test:** `src/__tests__/lint/live-flywheel.test.ts > live-flywheel canary
> lint-skill --ci --baseline`

**Status:** Real failure, reproducible at HEAD, fixable.

**Root cause:** I ran the exact command the test runs:
```
node dist/scripts/lint-skill.js --file ../skills/start/SKILL.md \
  --ci --baseline .lintskill-baseline.json --format json
```
The result is `{ summary: { errors: 1, warnings: 5 } }` but the test asserts
`parsed.summary.errors === 0`. The single error is:

- `AUQ003` at `skills/start/SKILL.md:421` — Question header "Discovery
  depth" is **15 chars**; AskUserQuestion UI truncates headers >12 chars.

Plus 5 warnings (4× `PLACE001` placeholder definitions missing for
`<detail>` and `<count>` in the "Display the welcome banner" step; 1×
`SLASH001` for `/xf` not resolving to any installed skill).

**Not a flaky test.** This is a product bug that shipped in skill
`skills/start/SKILL.md` during this wave's refinement commits
(`3011216`, `9a8b994`). The canary test is doing its job — the baseline
is correctly refusing to whitewash a new error.

**Fix options (not this review pass):**
1. Rename the AskUserQuestion header "Discovery depth" → ≤12 chars
   (e.g. "Depth").
2. Define `<detail>` / `<count>` placeholders in the welcome-banner step.
3. Replace `/xf` with a real skill ref (`SLASH001` suggests `/dsr`, `/ide`,
   `/slb`).

This wants a separate follow-up bead: "Fix SKILL.md lint errors in
skills/start/SKILL.md:421 & 169-184 & 427."
