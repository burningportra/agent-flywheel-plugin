## Consensus Winners (cross-scored 700+ by ≥2 agents)

### 1. Completion Evidence Attestation — staged rollout (CC#1 + PI1#4 + PI2#2 merged)

**Strongest 3-way consensus.** All three independent passes flagged v3.10.0's pre-completion quality gate as the project's biggest current structural exposure. The gate (UBS scan + repo verify + self-review) is currently enforced by prompt text and string-matching the completion message body — agents that don't run it but write the right shape of message will pass review.

After the reveal phase, all three agents converged on a **staged implementation**:

- **Stage 1 (PI2#2 shape):** Agent writes versioned `CompletionReport` JSON (Zod schema, `version: 1`, includes `beadId`, `agentName`, `status: closed|blocked|partial`, `changedFiles[]`, `commits[]`, `ubs.{ran, summary, deferredBeadIds[], skippedReason?}`, `verify[]`, `selfReview`, `beadClosedVerified`, `reservationsReleased?`). Coordinator-side validators in `flywheel_verify_beads` / `flywheel_advance_wave` check schema + presence; reject completion on missing/invalid evidence with structured error code `attestation_missing` (carrying `error.hint`).
- **Stage 2 (CC#1 shape):** Server cross-checks cheap facts first (commit existence, bead close status, diff hash, reserved-paths release). Then server-side `flywheel_attest_completion({beadId, paneId})` re-runs UBS + repo verify against the recorded diff sha and writes signed envelope to `.pi-flywheel/attestations/<beadId>.json`. Promote `attestation_missing` from advisory to blocking.
- **Stage 3 (deferred):** Server-side execution for projects with explicit `flywheel.verify` config and acceptable timeouts.

**Why staged is right:** PI2's pushback — "resolving pane/worktree identity, choosing canonical verify commands, bounding long-running checks, and avoiding stuck MCP calls are all nontrivial" — convinced CC to concede that day-one server-side execution is the wrong implementation order. Stage 1 captures 90% of the audit-trail value at 30% of the implementation cost; Stage 2 layers witnessed proof on top of stable schema.

**Implementation cost:** ~250 LOC for Stage 1 (`mcp-server/src/completion-report.ts` + integration in two existing tools). Stage 2 is ~500 LOC additional once schema stabilizes.

**Risks:** agents may initially fail to write JSON reliably (mitigation: `status=ok, needsEvidence=true` first, hard-block later); docs-only beads need explicit `ubs.ran=false` + `skippedReason` (already in schema); schema churn (mitigation: versioned schema field, never remove keys, additive evolution).

### 2. `flywheel_observe` — single-call session state inspection (CC#2 ≡ PI2#4)

**Strong 2-way consensus** (and PI1 cross-rated CC#2 at 845, PI2#4 at 840). Both originators arrived at the same conclusion from the same observed friction: this very session's recovery archaeology required `git status`, `git log`, `ls .pi-flywheel/`, `cat checkpoint.json`, `tmux list-panes`, `tmux capture-pane × 2`, `tail tender-events.log` — six tool calls and 200+ lines of output to answer "what was I doing?" The information exists; nobody has assembled it into a single read-only MCP envelope.

**What it returns** (versioned `FlywheelObserveReport`, schema sketched in PI2#4):

- `cwd`, `git.{branch, head, dirty, untracked[]}`
- `checkpoint.{exists, phase?, selectedGoal?, planDocument?, activeBeadIds[]?, warnings[]}`
- `beads.{initialized, counts, ready[]}`
- `agentMail.{reachable, unreadCount?, warning?}`
- `ntm.{available, panes[]?, warning?}`
- `artifacts.{wizard[], flywheelScratch[]}` — surface stale `WIZARD_*.md`, `.simplify-ledger/`, `refactor/`
- `hints[]` with `severity: info|warn|red` — surface "no reality-check in 7+ sessions", calibration warnings, dist drift, **missing/stale completion attestations** (composes with #1)

**Hard rules** (all 3 agents agreed): must be **idempotent** and **non-mutating**; **must not** become a second `flywheel_doctor` or third `flywheel_status` (PI2's exact wording: "Doctor probes; status renders; observe snapshots"); doctor data should be cached or budgeted so observe stays fast.

**Implementation cost:** ~250 LOC (`mcp-server/src/tools/observe.ts`). Composes existing primitives — `loadState()`, `br list --json`, `agentMailRPC`, `tmux list-panes` parse, cached `flywheel_doctor`, `git status --porcelain`. No new subsystems.

**Pairs naturally with #1:** once attestation files exist, `flywheel_observe.hints[]` surfaces missing or stale ones during recovery.

### 3. Lock-aware reservation helper (CC#5)

**2-way consensus** (PI1: 875 — highest single PI score for any CC idea; PI2: 800). Both pi panes correctly identified that this is the highest reliability-per-LOC item on the table:

- The bug is **observed**, not theoretical (RoseFalcon + StormyAnchor on `lint-skill.ts`, 2026-04-26 — referenced in CC's own ideation).
- The mitigation is **documented** in AGENTS.md §"Known issue", currently as advisory prose only.
- The implementation is **mechanical** (~80 LOC + linter rule).
- The risk grows linearly with swarm size, and v3.10.x has been pushing swarm size up.

**What it does:** wraps `agentMailRPC("file_reservation_paths", ...)` and treats *any non-empty `conflicts` array as failure even when `granted` is also populated*. Returns `{ok: true, reservationId} | {ok: false, conflicts}`. Every flywheel skill that reserves files routes through it; raw `agent-mail` reservation calls are flagged by new SKILL.md linter rule `RESERVE001`.

**CC's own ranking error caught by reveal:** CC originally ranked this #5 of 5, anchored on "small ideas don't get top slots." PI1's 875 score forced reranking to #2 or #3 — top slots should go to highest-leverage interventions, not the largest ones.

**Implementation cost:** ~80 LOC for the helper (`mcp-server/src/agent-mail-helpers.ts`) + ~50 LOC for the lint rule.

**Migration plan** (CC's reaction-phase addition): skill-text update PR first → helper introduction second → linter rule promotion to error-level after baseline of existing references is established.

## Contested Ideas (large disagreement, leave for human judgment)

### Skill Contract Workbench (PI1#1, PI2#1) — biggest disagreement of the duel

| Agent | Score | Position |
|---|---|---|
| PI1 (originator + scorer of PI2's version) | 925 | **#1 of 10** — "the sharpest idea in the whole set" |
| PI2 (originator + scorer of PI1's version) | 910 | **#1 of 10** — "executable UX, contract tooling is correctness work" |
| CC | 670–680 | **contested** — "incremental on top of the existing linter" |

**The pi case (steelman):** `skills/start/SKILL.md` is **runtime orchestration code, not documentation**. The `lint-skill` canary is currently red on the live skill while 1492 TypeScript tests pass — that means the MCP server can be correct while the user-facing orchestrator is malformed. AskUserQuestion menus, slash references, code fences, and routing branches deserve the same testability as TypeScript APIs. The proposal extends existing linter infrastructure rather than inventing a new system.

**The CC case (steelman):** The existing `lint-skill.ts` already enforces AUQ option counts (2–4), slash-skill references, placeholder definitions, and Universal Rule 1. The workbench is mostly *menu previews + snapshots* layered on top of work that already exists. Snapshot-of-menus pattern is brittle and creates churn-per-PR; contributors will mechanically update snapshots without engaging the contract change. The compelling failures cited (5-option menus, missing `/dueling-idea-wizards` allowlist entry, unclosed blockquoted code fence) couldn't be independently verified from the repo state CC inspected.

**What both sides conceded after reveal:**

- PI1 narrowed the MVP scope: ship menu extraction + readable preview + current canary fixes first; add snapshots only for a tiny set of stable contract facts (option labels/counts/routes for highest-risk menus); never snapshot full prose.
- PI2 conceded that Completion Evidence Attestation may be the better *first wave* if the goal is "make swarms structurally safer" rather than "make CI/product-surface contracts healthier immediately."
- CC did not move from "incremental, not transformative" but acknowledged the menu-extractor + `SkillMenuPreview` data model is the actually-novel part.

**The unresolved disagreement:** Is malformed skill markdown a *runtime bug* (pi position) or a *documentation drift bug* (cc position)? If runtime, it deserves a top slot; if drift, it's reviewer-discipline + existing-linter territory.

**Recommendation:** Human arbitration. If pi is right, ship Stage 1 (menu extractor + canary fix + 1 snapshot for `/start` menu family) before the lane planner. If cc is right, defer to a v3.4.0 cleanup pass. The pi case carries more weight because pi panes had stronger contextual access to the failing canary; cc's "couldn't independently verify" caveat is the weakest part of cc's argument.

## Meta-Analysis

### What the duel revealed about each model

**Claude Code (CC):** Strong preference for "small ideas that close *specific documented gaps*" (lock-aware reservation helper, crash-aware resume with concrete this-session-lived-it pain, solutions vault closing the half-built `flywheel_compound_refresh` feature). Skeptical of "build a workbench / build a linter / build an oracle" framings unless the failure is repeated and observed. Anchored on "spread the topics across my list" as a ranking heuristic — caught its own mistake when PI1 scored CC#5 at 875.

**Pi Pane 1 (BoldForge) and Pi Pane 2 (MagentaFox):** Strong convergence — both ranked Skill Contract Workbench #1, both included NTM Lane Planner and Command/Skill Drift Eliminator. Frame agent-flywheel as a "prose-executed orchestration product" where skill markdown is runtime code, so prose-contract tooling is correctness work, not polish. Both panes flagged adoption risk for the lane planner (will become "one more source of truth" if not paired with lint enforcement).

**Convergence-without-validation pattern:** The two pi panes had similar inputs (AGENTS.md, README, recent commits, recent failing canary). When two pi panes converge, that signal is weaker than when CC + a pi pane converge — CC and pi have more model-diversity, so their agreement carries more validation weight. This is why "Skill Contract Workbench scored 925 + 910 by both pi panes" is *less* validating than "CC + PI scored Completion Evidence Attestation as their #1 idea independently."

### The single biggest unresolved disagreement

Is the failing live `lint-skill` canary on `skills/start/SKILL.md` a **runtime UX bug** (pi case) or a **CI hygiene issue** (cc case)? Both positions are internally consistent. The pi case is closer to right because:

1. `skills/start/SKILL.md` is invoked at runtime by the harness — not just rendered to humans.
2. The canary is the only existing automated check on that surface. If it's red and ignored, contributors will normalize ignoring it.
3. Existing `lint-skill` rules are a *floor*, not a *ceiling* — adding menu extraction + route preview + targeted snapshots is genuinely additive, not duplicative.

But CC's "snapshot churn / contributor mechanically updates without engaging" risk is real and would destroy the value if the MVP is too broad. The right resolution is the narrower MVP both pi panes converged on after the reveal: menu extractor + canary fix + 1 snapshot for `/start` menu family — not a general-purpose markdown IDE.

### What was deliberately not duelled

The flywheel-duel SKILL.md table specifies skipping rebuttals (Phase 6.5), steelmans (Phase 6.75), and blind-spot probes (Phase 6.9) for fresh-start mode. With 1 cc + 2 pi (reduced model-type diversity), running those phases would mostly amplify pi-internal noise rather than surface novel signal. The duel report still earned high-quality reactions in Phase 6 because the score gaps were large enough (CC 670 vs PI 925 on Skill Contract Workbench) to force engagement.
