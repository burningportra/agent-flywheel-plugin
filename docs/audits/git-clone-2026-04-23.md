## Per-site analysis

### 1. `mcp-server/src/prompts.ts:researchInvestigatePrompt`

**Before:**

```bash
git clone --depth 1 ${externalUrl} ${cloneDir} 2>&1 || echo "Clone failed - repo may not exist or require auth"
```

* `externalUrl` is user-supplied via the `/flywheel-research` slash command (`$ARGUMENTS`).
* No host allowlist â€” a URL pointing at an attacker-controlled mirror (or a typo-squatted domain) is cloned without scrutiny.
* `--depth 1` and no `--single-branch` / `--no-tags` / `--recurse-submodules=no` means submodule and tag surface is wider than necessary.
* No SHA pinning â€” the proposal cites files and patterns without any way for a reader to reproduce the exact tree.
* Error handling uses `|| echo` which masks a non-zero exit from later steps in the same shell block.

**After this bead:**

* The emitted bash block now hard-fails (not just `echo`s) on non-allowlisted hosts and on clone failure.
* Uses `--single-branch --no-tags --recurse-submodules=no` and an explicit `--` separator.
* Captures `git rev-parse HEAD` and instructs the agent to surface it as a `Source:` line in the final proposal.
* A programmatic `safeClone` helper (`mcp-server/src/utils/clone-safety.ts`) is now available for any future caller that wants an in-process clone instead of emitting shell. It validates host, protocol, ref, and returns `{ path, head_sha, url, source }`.

### 2. `skills/flywheel-research/SKILL.md`

* Command-side guidance updated to make the clone-safety rules explicit and to require a `Source: <url> @ <short-sha>` header in the generated research doc. This is the user-visible contract that makes the SHA verifiable.

## Allowlist

Default allowed hosts (any one of these passes `validateCloneUrl`):

* `github.com`
* `gitlab.com`
* `bitbucket.org`
* `codeberg.org`
* `git.sr.ht`

Callers can pass `extraAllowedHosts` to support GitHub Enterprise or other self-hosted instances. This is intentionally NOT an env var â€” an env-var allowlist would recreate the CE mistake of letting a hostile `.envrc` redirect clones.

## Env-var policy

One (and only one) env var is honored by `clone-safety.ts`:

* `FLYWHEEL_ALLOW_INSECURE_CLONE=1` â€” allow `http://`, `git://`, `ssh://` URLs. Defaults off. This does NOT extend the host allowlist.

We deliberately did NOT add a `FLYWHEEL_CLONE_SOURCE` override. That is the exact footgun CE shipped (`COMPOUND_PLUGIN_GITHUB_SOURCE` overrides the install source with zero validation). If a user wants to clone from a private GHE host, they pass `extraAllowedHosts` programmatically or explicitly confirm the URL in the `/flywheel-research` command â€” never via env.

## Defense-in-depth checklist (applied)

* [x] Host allowlist before any network I/O.
* [x] Protocol allowlist (`https:` default; opt-in to `http:/git:/ssh:`).
* [x] Reject `file://`, `javascript:`, and arbitrary other schemes even with the insecure bypass.
* [x] Reject URLs starting with `-` (arg-masquerading).
* [x] Reject URLs containing newlines / NUL bytes (log-injection and arg-smuggling).
* [x] Reject embedded credentials in URLs (so they never land in logs).
* [x] Validate any branch/tag ref (reject `--â€¦`, whitespace, `..`, `:`).
* [x] Pass `--` separator to `git clone` as belt-and-suspenders.
* [x] `--depth 1 --single-branch --no-tags --recurse-submodules=no` by default.
* [x] Pin HEAD SHA via `git rev-parse HEAD` and return it to the caller.
* [x] Surface the pinned SHA in the `/flywheel-research` proposal (`Source:` line).
