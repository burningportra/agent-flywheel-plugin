# agent-flywheel v3.4.0 — Robustness-Focused Implementation Plan

**Author:** robustness perspective planner
**Drafting note:** Agent Mail bootstrap was unavailable while drafting in this
session, so this plan was produced locally against repo sources and will be
handed off with a final delivery retry at the end.
**Date:** 2026-04-21
**Coordinator:** LilacRidge
**Target release:** v3.4.0 observability bundle
**Primary goal:** ship observability features without introducing new silent
failure classes, cancellation leaks, unbounded memory growth, or cross-process
corruption.

---

## 0. Executive summary

v3.4.0 adds five subsystems that all touch operational edges rather than core
business logic:

1. `flywheel_doctor` — a one-shot dependency and health diagnostic MCP tool.
2. Shared-write hotspot reporting — a `plan-simulation.ts` extension that
   predicts wave-level file contention before launch.
3. Post-mortem CASS autogen — an `episodic-memory.ts` extension that turns a
   just-finished session into a bounded, reviewable post-mortem draft.
4. Bead-template library — hardening and plumbing for reusable bead templates
   through `bead-templates.ts` and `deep-plan.ts`.
5. Error-code telemetry — a bounded session-end aggregation path that persists
   error-code summaries to CASS with a local fallback.

The robustness posture for these features is:

- Read-only features should degrade to warnings rather than crash the session.
- Anything that writes persistent state must be bounded, atomic, and
  reconstructable from simpler primitives.
- New subprocess usage must thread `AbortSignal` through `ExecFn`.
- Mixed-version artifacts must be readable without forcing full migrations.
- Heuristics that can lie must surface confidence and must not silently drive
  irreversible behavior.
- Concurrent sessions on the same repo must not corrupt each other's
  telemetry, post-mortems, checkpoints, or launch decisions.

This plan assumes the current repo state documented in source:

- `mcp-server/src/server.ts` currently registers 8 primary tools.
- `mcp-server/src/errors.ts` exposes 16 `FlywheelErrorCode` values and the
  structured error envelope used at the MCP boundary.
- `mcp-server/src/exec.ts` already supports `{ timeout, cwd, signal }`.
- `mcp-server/src/cli-exec.ts` currently does not propagate `signal`, which is
  a relevant precondition for new retry-aware call sites.
- `mcp-server/src/plan-simulation.ts` is pure today and currently reasons only
  about conflicts, missing files, and execution order.
- `mcp-server/src/episodic-memory.ts` currently uses synchronous
  `execFileSync`, which is acceptable for legacy helpers but not for new
  cancellable session-end flows.
- `mcp-server/src/bead-templates.ts` already has validation and expansion
  helpers, but the planner pipeline uses templates only as prompt text.
- `mcp-server/src/checkpoint.ts` already gives us atomic write-rename behavior,
  `.corrupt` quarantine, and version/hash validation.
- `mcp-server/src/mutex.ts` is in-process only; cross-process coordination is
  expected to come from Agent Mail and file reservations.

The rest of this document is deliberately failure-first. The core deliverable
is the failure-mode matrix. The bead sketch and verification blocks are then
derived from that matrix rather than the other way around.

## 1. Grounding in current code and constraints

### 1.1 Files and seams that matter

- `mcp-server/src/server.ts`
  Adds the new `flywheel_doctor` tool to the primary tool list and default
  runners map. Also the likely place to add session-end telemetry flush hooks
  around the tool handler boundary.
- `mcp-server/src/errors.ts`
  The only acceptable home for new `FlywheelErrorCode` values and default
  retryability policy.
- `mcp-server/src/exec.ts`
  The base `ExecFn` contract already accepts `signal`.
- `mcp-server/src/cli-exec.ts`
  Retry wrappers currently pass `cwd` and `timeout` only. v3.4.0 robustness
  work must extend this to preserve cancellation semantics.
- `mcp-server/src/plan-simulation.ts`
  The existing pure engine for order, parallel groups, file conflicts, and
  missing file references. Hotspot logic should stay mostly pure here.
- `mcp-server/src/beads.ts`
  `extractArtifacts()` is the current file-path extraction heuristic, which is
  relevant for hotspot precision/recall tradeoffs.
- `mcp-server/src/episodic-memory.ts`
  The existing memory helper module; the new post-mortem autogen should extend
  it with async, bounded, non-dumping behavior rather than bolting into tools.
- `mcp-server/src/bead-templates.ts`
  Existing template integrity checks, expansion rules, and prompt-formatting
  helpers live here.
- `mcp-server/src/deep-plan.ts`
  The current deep-planner entry point and the natural seam for carrying
  template metadata forward in a robust way.
- `mcp-server/src/tools/approve.ts`
  Where hotspot output needs to influence launch decisions without making the
  tool nondeterministic or unbounded.
- `mcp-server/src/tools/review.ts`
  Important regression surface because prior sessions saw already-closed bead
  parse issues around `br show`. v3.4.0 should not re-open this edge.
- `mcp-server/src/state.ts` and `mcp-server/src/checkpoint.ts`
  The source of truth for crash recovery and partial-state behavior.

### 1.2 Hard constraints we must preserve

- No `console.log` in MCP server code.
- No direct edits to `mcp-server/dist/`.
- TypeScript strict mode remains non-negotiable.
- Relative imports must keep `.js` extensions.
- All new subprocess calls must use `ExecFn` with `timeout`.
- All new subprocess calls must pass `signal` when a caller received one.
- No direct writes to `.pi-flywheel/checkpoint.json`.
- `dist/` remains committed, so the plan must assume a build + dist refresh
  bead exists before merge, even though this planning task does not run builds.

### 1.3 Prior-session context that must be explicitly addressed

- Gemini CLI is often absent, so `ntm --gmi` must be treated as optional and
  doctor must report rather than crash.
- Agent Mail `contacts_only` blocked planner-to-coordinator first contact.
  Coordinator bootstrap must auto-set `contact_policy=auto`.
- `flywheel_review` previously errored on already-closed beads because of
  parse assumptions on `br show` output.
- `ntm send --pane=<int>` is the only supported form; pane names are wrong.
- Monitor tools can exit non-zero on noisy iterations; templates must use
  `|| true` where the tool is informational rather than authoritative.
- Checkpoints can be corrupt or stale; post-mortem must reconstruct rather than
  die if feasible.
- CASS can return very large payloads; post-mortem and telemetry must summarize
  rather than dump.

## 2. Robustness design principles

### 2.1 Degrade optional dependencies, block only on required invariants

`bv`, `cm`, Gemini, and Agent Mail availability should not all have equal
weight. Missing `br` is a blocking condition for bead-centric workflows.
Missing `bv` is not. Missing `cm` should degrade post-mortem storage to a local
draft rather than aborting session shutdown.

### 2.2 Keep heuristics advisory unless confidence is high

Hotspot detection is necessarily heuristic because it infers future writes from
bead descriptions. It may recommend a safer launch mode, but it should not
silently force an irreversible branch without surfacing confidence and reasons.

### 2.3 Preserve determinism at every pure boundary

If the same bead graph is analyzed twice, with the same inputs in a different
array order, the hotspot matrix and recommendations must be byte-for-byte
stable after normalization. Non-determinism at planning time causes false
debates, flaky tests, and impossible-to-compare review results.

### 2.4 Bound everything that can grow

Telemetry can easily hit 10k+ events in a noisy loop. CASS responses can be
large. Post-mortem drafts can accrete junk if not bounded. All new features
must define caps up front rather than after the first incident.

### 2.5 Writes must be atomic, reconstructable, and idempotent

Any v3.4.0 persistent write must satisfy at least one of:

- write-to-temp then rename,
- write to a session-scoped file with unique identity,
- be fully reconstructable from git, beads, and checkpoint state.

### 2.6 Cancellation is a feature, not an afterthought

The repo already chose `AbortSignal` at the `ExecFn` layer. New code must not
fall back to `execFileSync` or retry loops that ignore a cancelled parent.

### 2.7 Distinguish warnings from structured errors

Warnings are for degraded but survivable conditions.
Structured errors are for conditions the orchestrator needs to branch on.
Returning a warning-looking string where the orchestrator expects a code is a
robustness bug, not a UX nit.

## 3. Failure-mode enumeration (CORE deliverable)

The sections below enumerate subsystem-specific failures. Every entry includes:

- `Scenario` — the concrete precondition.
- `Detection` — the signal or mechanism used.
- `Response` — structured error code, warning, retry, fallback, or no-op.
- `User-facing effect` — abort vs continue.

### 3.1 `flywheel_doctor`

#### D1. Required `br` CLI is missing or not executable

- Scenario: the repo is in bead mode, but `br --version` fails with ENOENT or
  EACCES.
- Detection: `exec('br', ['--version'], { timeout, cwd, signal })` throws or
  returns non-zero; `classifyExecError` or a dedicated doctor classifier tags
  the failure.
- Response: mark the check `red`; return a structured
  `doctor_blocking_failure` result if doctor was called directly; no retry on
  ENOENT/EACCES; include remediation to install or fix permissions.
- User-facing effect: the doctor completes with a red row; the session does
  not auto-launch bead workflows until the user fixes the dependency.

#### D2. Optional `bv` CLI is absent

- Scenario: `bv` is not installed, but the rest of the stack is healthy.
- Detection: `exec('bv', ['--help'], ...)` returns ENOENT or non-zero.
- Response: mark `yellow`; do not throw; explain that graph-theoretic insight
  and some verify/triage enhancements degrade to simpler logic.
- User-facing effect: session continues; doctor reports degraded capability.

#### D3. `ntm` exists but Gemini-backed mode is unavailable

- Scenario: `ntm` is installed, but `ntm --gmi` or the Gemini path it would
  rely on is unavailable, which has been seen in prior sessions.
- Detection: doctor runs a conservative `ntm --help` or version check and, if
  configured to examine Gemini support, treats any `--gmi` failure as an
  optional subcheck rather than a fatal command.
- Response: mark `yellow`; explicitly state "NTM available; Gemini optional
  path unavailable"; never call a failing `--gmi` path during the doctor’s
  default green sweep.
- User-facing effect: session continues; swarm templates and runbooks advise
  non-Gemini paths unless the user explicitly opts in.

#### D4. Agent Mail HTTP transport is unreachable

- Scenario: the stable transport should be `http://127.0.0.1:8765/mcp`, but the
  server is not listening or the endpoint refuses connection.
- Detection: doctor performs a short HEAD or GET check through a signal-aware
  `curl` exec with a sub-second timeout.
- Response: mark `yellow` or `red` depending on current phase intent:
  `yellow` during generic inspection, `red` if the caller is about to launch a
  swarm; no blind retries beyond one short retry; suggest single-agent or
  worktree fallback.
- User-facing effect: session continues in degraded mode unless the user
  insists on swarm launch, in which case the orchestrator asks before
  proceeding.

#### D5. Coordinator bootstrap would hit `contacts_only`

- Scenario: the coordinator is registered with Agent Mail, but its contact
  policy is still `contacts_only`, blocking first contact from planners.
- Detection: bootstrap logic or a doctor subcheck inspects contact policy or
  observes a blocked DM attempt in a mocked health path.
- Response: for coordinator sessions, `macro_start_session` should
  opportunistically set `contact_policy=auto`; doctor reports `yellow` if the
  policy is still restrictive after bootstrap.
- User-facing effect: session continues; first-contact failures are prevented
  rather than discovered late at handoff time.

#### D6. `cm` / CASS is missing

- Scenario: `cm --version` fails because the CLI is not installed.
- Detection: direct version exec returns ENOENT/non-zero.
- Response: mark `yellow`; do not crash; note that post-mortem storage and
  telemetry persistence will use local fallbacks.
- User-facing effect: session continues; only memory-backed conveniences
  degrade.

#### D7. Doctor parser sees noisy or malformed command output

- Scenario: a dependency emits banners, ANSI noise, or malformed JSON that
  cannot be parsed into the expected row shape.
- Detection: schema parse or parser helper fails for one row while the process
  itself returned code 0.
- Response: mark that row `yellow` with `parse_failure` details; preserve raw
  preview in structured `details`; continue remaining checks.
- User-facing effect: session continues; doctor reports that the check result
  is advisory because parsing was degraded.

#### D8. One dependency check hangs

- Scenario: a binary exists but stalls, e.g. `cm` or `ntm` blocks on startup.
- Detection: per-check timeout fires; the exec promise rejects with a timeout.
- Response: classify the row as `red` or `yellow` depending on whether the
  dependency is required; retry exactly once for network-shaped checks only;
  otherwise record `exec_timeout`.
- User-facing effect: session continues with a visible timeout row; doctor
  still returns within the global sweep budget.

#### D9. User presses Ctrl-C mid-doctor

- Scenario: the user cancels a full sweep while some checks have settled and
  others are in flight.
- Detection: parent `AbortSignal` is aborted; in-flight `exec` calls reject
  with `AbortError` or equivalent.
- Response: gather all settled rows, mark unresolved rows as `cancelled`, and
  return an `exec_aborted` structured error with `details.partialReport`.
- User-facing effect: the direct doctor call stops early but still yields a
  partial table instead of disappearing without explanation.

#### D10. Two flywheel sessions inspect the same repo simultaneously

- Scenario: two agent sessions run `flywheel_doctor` against the same `cwd`.
- Detection: doctor sees an active checkpoint phase, Agent Mail reservations,
  build slots, or both.
- Response: doctor never acquires an exclusive lock; it reports `yellow` and
  identifies the competing session or reservation holder if known.
- User-facing effect: both sessions continue; doctor does not become a new
  source of contention.

#### D11. Checkpoint exists but is corrupt

- Scenario: `.pi-flywheel/checkpoint.json` exists but parse or hash validation
  fails.
- Detection: use existing `readCheckpoint()` behavior, which already returns
  `null` after moving corrupt content aside.
- Response: doctor marks the checkpoint row `yellow` with explicit text that
  recovery will rely on git and beads rather than checkpoint state.
- User-facing effect: session continues; the user sees that crash recovery is
  degraded before they need it.

#### D12. Dist-drift heuristic is stale or ambiguous

- Scenario: source files changed but no authoritative build has been run, and a
  fast doctor heuristic cannot prove whether `dist/` is actually out of sync.
- Detection: the heuristic notices `src/` touched without matching `dist/`
  touches, or sees a dirty worktree in only one side.
- Response: mark `yellow`, never `green` on ambiguity; reserve `red` for cases
  where a fast proof exists; recommend a build bead rather than running the
  build inside doctor.
- User-facing effect: session continues; doctor warns that CI may later fail
  `dist-drift` even though the local check was inconclusive.

### 3.2 Shared-write hotspot report

#### H1. Beads do not contain machine-extractable file scope

- Scenario: bead descriptions omit `### Files:` and also avoid path-like bullet
  lines, so `extractArtifacts()` yields little or nothing.
- Detection: hotspot engine sees many beads with empty artifact lists and a low
  extraction-confidence score.
- Response: return a `yellow` advisory or a direct
  `hotspot_input_unreliable` error when called standalone; do not force swarm
  recommendations from empty inputs.
- User-facing effect: approve flow continues, but launch advice falls back to
  existing conservative choices.

#### H2. Prose mentions a file path that is not actually written

- Scenario: a bead rationale says "keep behavior consistent with src/foo.ts"
  even though the bead will not edit that file.
- Detection: the parser extracts the path from free prose outside the files
  section and tags it low-confidence.
- Response: low-confidence paths contribute warnings only and cannot by
  themselves trigger a `high` contention recommendation.
- User-facing effect: session continues; false positives are visible but do not
  silently force coordinator-serial mode.

#### H3. Nonstandard path layouts cause false negatives

- Scenario: a bead writes `package.json`, `.github/workflows/ci.yml`, or some
  other file outside the current `extractArtifacts()` prefix set.
- Detection: path normalization and files-section parsing notice root-level or
  dot-prefixed paths not covered by the old prefix matcher.
- Response: extend hotspot extraction to prefer `### Files:` over prefix rules
  and normalize any path containing a slash or known filename pattern.
- User-facing effect: session continues; accuracy improves without breaking old
  beads.

#### H4. Matrix output changes with input order

- Scenario: the same beads arrive in a different array order because `br list`
  or caches changed ordering.
- Detection: determinism tests reverse the bead list and compare matrix output.
- Response: sort bead IDs before iteration, sort file lists, sort output rows by
  severity then file name, and sort row bead IDs alphabetically.
- User-facing effect: no visible flake; repeated runs yield stable output.

#### H5. Hotspot analysis races with bead edits

- Scenario: a second process modifies bead descriptions or dependencies while
  hotspot analysis is mid-computation.
- Detection: the analysis sees mismatched counts between the initial snapshot
  and a final quick consistency check, or approval detects a changed bead cache
  generation.
- Response: treat the analysis as a snapshot; if it is stale by the end of the
  approve call, mark the matrix `yellow` and recommend rerunning approval.
- User-facing effect: session continues; the tool never mixes two incompatible
  snapshots into one confident recommendation.

#### H6. Session cache returns stale bead content

- Scenario: `beads.ts` session cache still holds old bead content for up to 30s
  while a new approval cycle expects fresh descriptions.
- Detection: approval knows it just created or mutated beads and can invalidate
  caches, or a freshness marker on the snapshot indicates age > 0.
- Response: explicitly invalidate relevant caches before hotspot analysis on
  approval boundaries; standalone hotspot calls bypass or refresh cached data.
- User-facing effect: session continues; hotspot advice reflects current bead
  text rather than recently replaced drafts.

#### H7. Large waves make analysis expensive

- Scenario: a plan produces dozens or hundreds of beads, each with multiple
  paths, making naïve pairwise conflict scoring too expensive.
- Detection: input sizes exceed target thresholds such as 100 beads or 2k file
  mentions.
- Response: use a file-to-beads index rather than O(n²) comparison, cap detail
  rows to top-N, and emit a summary-only warning if bounds are exceeded.
- User-facing effect: approve flow continues within time limits; the user sees
  a bounded summary rather than a giant matrix dump.

#### H8. Path normalization mismatches collapse or split rows incorrectly

- Scenario: one bead lists `./src/foo.ts`, another lists `src/foo.ts`, and a
  third lists `src//foo.ts`.
- Detection: a normalization pass compares raw and canonical forms and notices
  duplicates collapsing into the same canonical path.
- Response: normalize leading `./`, repeated separators, and whitespace; retain
  raw path samples in debug details for traceability.
- User-facing effect: session continues; users see one correct row rather than
  multiple misleading pseudo-paths.

#### H9. Critical shared files are underweighted

- Scenario: `mcp-server/src/server.ts`, `types.ts`, `errors.ts`, or
  `checkpoint.ts` are touched by multiple beads but the simple count threshold
  labels them only medium.
- Detection: severity logic includes a critical-file allowlist layered on top of
  raw write count.
- Response: upgrade severity to `high` for duplicates on critical files, even
  if only two beads are involved.
- User-facing effect: approve flow becomes more conservative where it matters
  most and avoids optimistic swarm advice around shared contracts.

#### H10. Missing dependency edges hide a real parallel hazard

- Scenario: two beads touch the same file and should be serialized, but the
  dependency graph is incomplete so both appear in the same ready wave.
- Detection: hotspot analysis cross-checks wave grouping with file overlap and
  identifies a hazard even when base simulation still says the graph is valid.
- Response: mark the entry `high`, recommend coordinator-serial for those beads,
  and surface a follow-up suggestion to add a dependency edge.
- User-facing effect: session continues safely; the hazard becomes visible
  before agent launch.

#### H11. Dependency data itself is incomplete or unparsable

- Scenario: the dep map cannot be built because `br dep list` or equivalent
  parsing fails.
- Detection: dep-collection helper returns parse or CLI failure for the current
  snapshot.
- Response: degrade to a file-only matrix with an explicit warning that
  dependency-aware wave reasoning is unavailable; never claim `green` in this
  state.
- User-facing effect: session continues; the launch recommendation is more
  conservative and lower confidence.

#### H12. Human readers over-trust a heuristic recommendation

- Scenario: users interpret "swarm safe" as a guarantee instead of a heuristic
  recommendation derived from bead descriptions.
- Detection: not a machine signal; this is a product-risk identified in review.
- Response: every matrix includes a confidence note and says "recommended mode"
  rather than "safe/unsafe"; high-confidence only when all contested paths come
  from `### Files:` sections.
- User-facing effect: session continues with clearer framing, reducing misuse of
  heuristic output.

### 3.3 Post-mortem CASS autogen

#### P1. No checkpoint exists at wrap-up time

- Scenario: `saveState()` previously returned `false`, or the session ran
  without a persisted checkpoint.
- Detection: `readCheckpoint()` returns `null` and no valid checkpoint file is
  found.
- Response: do not abort; reconstruct a minimal session summary from git log,
  bead state, and tool outputs; tag the draft with a warning that checkpoint
  state was unavailable.
- User-facing effect: session continues; post-mortem quality is lower but a
  draft still exists.

#### P2. Checkpoint JSON is truncated or fails hash validation

- Scenario: `.pi-flywheel/checkpoint.json` was partially written or tampered
  with and is moved aside as `.corrupt`.
- Detection: existing `readCheckpoint()` parse/hash validation path returns
  `null` after quarantining the file.
- Response: emit `postmortem_checkpoint_invalid` only for direct post-mortem
  tool calls; otherwise reconstruct from git and note the corruption in the
  draft header.
- User-facing effect: session continues; the user sees an explicit corruption
  warning instead of a silent omission.

#### P3. Checkpoint is stale relative to current git HEAD

- Scenario: the checkpoint’s `gitHead` or timestamp no longer matches the
  current repo state because more commits landed after the last checkpoint.
- Detection: compare `git rev-parse HEAD` to checkpoint metadata and compare age
  to a staleness threshold.
- Response: include both checkpoint-derived and git-derived facts, prefer git
  for landed commit summaries, and add a draft warning about possible drift.
- User-facing effect: session continues; the post-mortem is labeled partial
  rather than pretending full accuracy.

#### P4. `br` is unavailable during post-mortem generation

- Scenario: post-mortem wants bead completion data but `br` is not installed or
  errors.
- Detection: `exec('br', ...)` returns ENOENT or non-zero.
- Response: fall back to checkpoint `activeBeadIds`, `beadResults`, and commit
  messages that mention bead IDs; do not block draft generation.
- User-facing effect: session continues; bead summaries may be less complete.

#### P5. `git` data is unavailable or shallow

- Scenario: `git log` fails, the repo is detached, or history is too shallow to
  find the relevant commit range.
- Detection: `git log` or `git rev-parse` returns non-zero.
- Response: do not abort; use checkpoint and in-memory tool results only; warn
  that landed commit evidence could not be collected.
- User-facing effect: session continues with a weaker factual basis.

#### P6. `cm` / CASS is missing at flush time

- Scenario: the autogen draft exists, but the storage backend for durable
  memory is unavailable.
- Detection: `cm --version` or `cm add` fails.
- Response: write the markdown draft to `docs/sessions/` and skip CASS store;
  record a local warning and return success-with-fallback.
- User-facing effect: session continues; the user still gets a local artifact.

#### P7. CASS returns or accepts enormous payloads

- Scenario: `cm context` or a related query yields a huge response, or a raw
  draft is too large to be safely stored.
- Detection: byte length exceeds configured input or output caps before store.
- Response: summarize to bounded sections, keep only top-N bullets and snippets,
  and never dump raw CASS payloads into the draft or telemetry.
- User-facing effect: session continues; post-mortem remains readable and
  bounded instead of exploding in size.

#### P8. Agent Mail data is unavailable for highlights

- Scenario: the post-mortem wants inbox or thread highlights, but Agent Mail is
  unreachable or not configured.
- Detection: agent-mail health or fetch fails during the optional highlights
  substep.
- Response: omit the highlights section and replace it with a one-line warning;
  do not fail the whole draft.
- User-facing effect: session continues; the draft is slightly less rich.

#### P9. Draft generation is cancelled mid-flight

- Scenario: user hits Ctrl-C while the draft is collecting git, bead, and CASS
  inputs.
- Detection: parent `AbortSignal` aborts; async execs reject with abort.
- Response: cancel remaining subprocesses, keep any already-built partial
  sections in memory, and if a local file write has not started, skip writes
  entirely rather than creating a half-finished markdown file.
- User-facing effect: operation aborts cleanly; no corrupted draft file is
  emitted.

#### P10. Duplicate drafts are generated for one session

- Scenario: post-mortem autogen is triggered twice on the same logical session,
  or the user reruns it after reviewing the first draft.
- Detection: session ID and target file path already exist with matching
  metadata.
- Response: make generation idempotent by overwriting only when the same
  session ID is requested and the file contains the autogen marker; otherwise
  suffix with `-r2`, `-r3`, and so on.
- User-facing effect: session continues; no silent clobbering of prior notes.

#### P11. Two sessions crash and generate drafts simultaneously

- Scenario: two independent sessions in the same repo both die and both try to
  emit a local post-mortem draft.
- Detection: target file naming would collide unless session identity is part of
  the path.
- Response: include session ID in filenames, write atomically with temp+rename,
  and never use a date-only filename as the sole key.
- User-facing effect: both sessions preserve their own artifacts without
  overwriting each other.

#### P12. `docs/sessions/` grows unbounded

- Scenario: months of sessions leave 100+ or 1000+ draft files in the repo.
- Detection: before writing, count existing autogen files and their aggregate
  byte size.
- Response: warn once above the soft cap, never auto-delete by default, and add
  a cleanup recommendation or an explicit maintenance bead rather than silently
  pruning history.
- User-facing effect: session continues; disk pressure is visible and
  actionable.

### 3.4 Bead-template library

#### T1. Planner references an unknown template ID

- Scenario: a deep plan or manual bead draft asks for a template that is no
  longer present in `bead-templates.ts`.
- Detection: `getTemplateById()` returns `undefined` or expansion fails at the
  lookup stage.
- Response: return `template_expansion_failed` or `not_found` at the direct
  expansion boundary; in planner flows, fall back to a fully custom bead
  description rather than leaking a raw template marker.
- User-facing effect: session continues; the bead still gets created, but with
  a fallback path or an explicit error if the user requested direct expansion.

#### T2. Required placeholders are missing

- Scenario: the chosen template needs `{{paneId}}` or `{{implementationFile}}`,
  but the planner failed to supply one or more values.
- Detection: existing `expandTemplate()` required-placeholder validation finds
  blank or absent fields.
- Response: return `template_expansion_failed` with the missing names and known
  placeholder list; never emit partially expanded template text.
- User-facing effect: bead creation pauses for that template path, but the
  broader planning session can continue with correction or fallback.

#### T3. Unresolved placeholders leak into final bead text

- Scenario: a template expands with `{{placeholder}}` still present because of a
  mismatch between the template body and placeholder definitions.
- Detection: existing unresolved-placeholder check fires after replacement.
- Response: treat as a hard expansion failure; never let `br create` see raw
  template markers; add a regression test for the affected template.
- User-facing effect: session continues only after correction; this should not
  become a hidden degraded mode.

#### T4. Template version changes after a plan references it

- Scenario: a plan was synthesized against template version 1, but by the time
  beads are created the template is on version 2 with different placeholders or
  semantics.
- Detection: plan metadata includes `templateId@version`; expansion compares the
  pinned version to the currently loaded version.
- Response: return `template_version_conflict` or fall back to the pinned,
  stored copy if available; never silently reinterpret the plan under a new
  template version.
- User-facing effect: session pauses for the affected bead or uses the pinned
  copy; there is no silent semantic drift.

#### T5. Spawn-swarm template uses a pane name instead of integer `--pane`

- Scenario: a template or example instructs `ntm send --pane=backend` even
  though only integer pane IDs work.
- Detection: template lint or tests assert that the spawn-swarm template uses a
  numeric placeholder and example.
- Response: fail template validation or tests; correct the template before
  release; include explicit wording that pane IDs are integers only.
- User-facing effect: no broken command reaches users if validation catches it;
  otherwise, the failure is deterministic and immediately actionable.

#### T6. Informational monitor commands exit non-zero on noise

- Scenario: a template includes a monitor command that is informative, but the
  tool exits non-zero during routine noisy iterations.
- Detection: prior-session regression tests and template example review catch
  the command shape.
- Response: template examples and generated commands append `|| true` for
  monitor-only paths, while retaining strict behavior for commands that gate
  correctness.
- User-facing effect: session continues; users do not see false-failure noise
  from purely observational commands.

#### T7. Template `filePatterns` are too broad or stale

- Scenario: a template declares `src/**/*.ts` when it really targets a much
  narrower slice, producing poor hotspot and dependency hints.
- Detection: integrity and example-review tests compare patterns to examples and
  flag wildcards that are broader than the template’s own examples justify.
- Response: treat as a validation warning at load time and a review-blocking
  issue for new templates.
- User-facing effect: session continues, but the template library is corrected
  before merge rather than after bad planning output accumulates.

#### T8. Raw template markers leak into saved beads

- Scenario: planning output says `[Use template: spawn-swarm]` or `see template`
  and the bead creation path fails to expand it before `br create`.
- Detection: existing bead hygiene checks in `beads.ts` already detect raw
  template markers and unresolved placeholders.
- Response: keep those validations strict and add direct regression coverage for
  the new observability templates; fail approval until raw markers are removed.
- User-facing effect: the user sees a concrete quality failure instead of a
  silently malformed bead graph.

#### T9. Template examples drift from actual command semantics

- Scenario: examples in the library still show old flags, wrong paths, or stale
  workflow advice even though validators pass.
- Detection: example-based golden tests and fresh-eyes review gates compare
  examples to current command contracts.
- Response: examples are treated as testable artifacts, not comments; failing
  examples block the merge.
- User-facing effect: users and future agents see trustworthy examples.

#### T10. Template integrity warnings are ignored

- Scenario: `TEMPLATE_INTEGRITY_WARNINGS` records issues, but the planner prompt
  still formats and exposes the broken template.
- Detection: boot-time or test-time assertions check that integrity warnings are
  empty for release builds.
- Response: fail CI or at minimum fail the template-loading bead; do not expose
  a broken template library to planners.
- User-facing effect: the release is blocked until template hygiene is fixed.

#### T11. Deep-plan plumbing overfits templates and harms generality

- Scenario: planners start treating templates as mandatory, leading to awkward
  or forced fits for uncommon tasks.
- Detection: review gates observe plans that force template IDs into beads that
  are better written custom.
- Response: keep the prompt language explicit that templates are optional
  accelerators; add template confidence or suitability guidance instead of a
  hard requirement.
- User-facing effect: session continues with better planning quality; custom
  beads remain first-class.

#### T12. Template schema additions break older consumers

- Scenario: new required fields such as `version` or `testStrategy` are added to
  `BeadTemplate`, but some formatter or prompt consumer still assumes the older
  shape.
- Detection: strict TypeScript compilation and unit tests around
  `formatTemplatesForPrompt()` and `listBeadTemplates()`.
- Response: introduce additive fields with defaults, version the template
  object, and keep prompt formatting tolerant where possible.
- User-facing effect: session continues; older flows either ignore the extra
  metadata or explicitly surface version conflicts.

### 3.5 Error-code telemetry

#### E1. Error event volume grows without bound

- Scenario: a noisy 4-hour session emits 10k+ structured errors.
- Detection: in-memory aggregator notices total event count crossing threshold
  bands.
- Response: store only per-code counters plus a bounded ring buffer of recent
  samples; drop full per-event history after cap and increment a
  `droppedSamples` counter.
- User-facing effect: session continues; telemetry remains useful without
  ballooning memory.

#### E2. One logical failure gets counted multiple times

- Scenario: the same failure is classified in `classifyExecError`, wrapped again
  in `makeFlywheelErrorResult`, and maybe logged at the tool boundary too.
- Detection: telemetry integration points are reviewed for duplicate counting on
  the same error path.
- Response: count only at the structured-envelope boundary and in top-level
  uncaught-error handling; never count on both helper classification and final
  wrapping.
- User-facing effect: session continues; telemetry reflects real frequency
  rather than instrumentation noise.

#### E3. Telemetry counts its own storage failures recursively

- Scenario: `cm add` fails during telemetry flush, and the telemetry system
  tries to record that failure into itself again.
- Detection: flush path knows it is inside telemetry persistence and can guard
  against re-entrant counting.
- Response: use a re-entrancy flag; if telemetry flush fails, write one fallback
  warning and do not recursively emit more telemetry events.
- User-facing effect: session continues; failure is visible once, not amplified.

#### E4. Session exits before flush completes

- Scenario: the process crashes or the user aborts during session shutdown.
- Detection: process-level shutdown or tool-finally hooks start flush but do not
  get enough time to complete the CASS path.
- Response: prefer a fast local fallback first, then best-effort CASS store;
  if both cannot finish, keep the in-memory aggregate out of the critical path.
- User-facing effect: session ends cleanly; telemetry may degrade to local-only
  rather than blocking shutdown.

#### E5. CASS store fails

- Scenario: `cm add` returns non-zero, times out, or is unavailable at session
  end.
- Detection: signal-aware `exec('cm', ...)` with direct status checking.
- Response: emit `telemetry_store_failed`, persist `.pi-flywheel/error-counts`
  fallback, and continue shutdown.
- User-facing effect: session continues; telemetry durability degrades locally.

#### E6. Two sessions crash and flush telemetry simultaneously

- Scenario: both sessions try to update the same fallback telemetry state in the
  same repo at once.
- Detection: merge logic sees file-version or write-race anomalies, or a
  per-session spool file already exists for another session.
- Response: use session-scoped spool data and a merged summary file; on merge
  conflicts, retry twice and then leave both spool files for later compaction.
- User-facing effect: session continues; worst case is delayed compaction, not
  data loss.

#### E7. Fallback telemetry JSON is corrupt

- Scenario: `.pi-flywheel/error-counts.json` is truncated or malformed due to a
  previous interrupted write.
- Detection: JSON parse fails before merge.
- Response: move the corrupt file aside, reconstruct from any per-session spool
  files that still exist, or start a new summary file if reconstruction inputs
  are absent.
- User-facing effect: session continues; telemetry history may lose some detail
  but current-session counts still persist.

#### E8. Mixed-version telemetry schemas coexist in CASS

- Scenario: v3.4.0 writes one shape and v3.5.0 later writes another.
- Detection: reader sees `schemaVersion` or missing fields that do not align
  with the current schema.
- Response: only aggregate on stable common fields (`sessionId`, `code`,
  `count`, timestamps) and ignore unknown fields; never fail the entire
  reporting path because one entry is newer.
- User-facing effect: session continues; older and newer entries coexist.

#### E9. Error `details` payloads are too large or sensitive

- Scenario: a structured error includes huge stderr blobs or path-rich details
  that should not be mirrored into telemetry.
- Detection: before recording, telemetry sanitizer enforces per-field byte caps
  and allowlists only safe keys for samples.
- Response: counters keep counts; samples keep only trimmed `tool`, `phase`,
  `code`, timestamp, and a redacted preview if safe.
- User-facing effect: session continues; telemetry stays bounded and lower risk.

#### E10. Rare but severe codes disappear from top-N summaries

- Scenario: one `internal_error` happens once, while a noisy but harmless code
  happens hundreds of times, hiding the severe event from `topN`.
- Detection: summary generation sees low-frequency high-severity codes outside
  the top count ranking.
- Response: produce two summary slices: top by count and top by severity, or
  always include one sample of severe classes even if rare.
- User-facing effect: session continues; serious rare failures stay visible.

#### E11. Resumed sessions double-count prior errors

- Scenario: a session resumes from checkpoint and telemetry memory is rebuilt
  without session identity, causing old counts to be counted again.
- Detection: session-scoped telemetry has the same session ID and finds an
  existing in-progress aggregate.
- Response: tie counts to a session ID and persist monotonic sequence numbers or
  "already flushed" flags; merging a resumed session updates instead of cloning.
- User-facing effect: session continues; telemetry reflects one logical session.

#### E12. Telemetry work slows down hot paths

- Scenario: instrumentation itself adds noticeable latency to every structured
  error emission.
- Detection: bounded benchmark tests compare tool error paths with telemetry
  enabled vs disabled.
- Response: telemetry increments must be O(1), allocate minimally, and defer any
  serialization until flush time.
- User-facing effect: session continues with no observable slowdown on normal
  error handling.

## 4. Cross-process concurrency

### 4.1 Same repo, two flywheel sessions

Decision:

- `flywheel_doctor` is read-only and never takes an exclusive lock.
- Hotspot analysis is snapshot-based and advisory only.
- Post-mortem and telemetry writes are session-scoped and atomic.
- Launch decisions must consult current Agent Mail health if swarm launch is
  about to happen, not the stale state from an earlier snapshot.

Who wins on lock contention:

- Doctor: nobody "wins"; both report current contention and continue.
- Hotspot analysis: nobody "wins"; each analysis is bound to its own bead
  snapshot and does not mutate shared state.
- Telemetry fallback writes: neither should lose data; per-session spool files
  must survive even if summary merge is contended.
- Post-mortem draft writes: filenames must include session ID so both succeed.

### 4.2 Doctor and launch gating

When two sessions are active on one repo:

- Doctor reports `yellow`, not `red`, unless the current action is about to
  spawn parallel agents into the same checkout without Agent Mail reservations.
- Approval or launch logic, not doctor, decides whether to block swarm launch.
- If Agent Mail is healthy, file reservations remain the cross-process source of
  truth.
- If Agent Mail is unhealthy, approval should bias toward coordinator-serial or
  worktree isolation rather than optimistic swarm execution.

### 4.3 Telemetry aggregation when two sessions crash simultaneously

Robust design choice:

- Use a session ID for every telemetry aggregate.
- Write a per-session spool record first.
- Merge into `.pi-flywheel/error-counts.json` second.
- If merge fails, keep the spool record and report the merge as degraded rather
  than losing the session’s data.

This avoids a fragile "single shared JSON file is the only truth" design.

### 4.4 Agent Mail outage mid-session, right before swarm launch

This is a concrete required case from the task.

Decision:

- `flywheel_approve_beads` must re-check Agent Mail health at the point of
  launch if the chosen mode depends on reservations or DM coordination.
- If health is down, it should not proceed as though reservations exist.
- Safe fallback order:
  1. coordinator-serial in the same branch if the user wants to continue fast,
  2. worktree isolation if setup exists and the user accepts the heavier path,
  3. stop and ask for remediation if the plan explicitly requires swarm.

Behavior:

- direct launch becomes a user-visible choice,
- not a hidden downgrade,
- and not a crash.

### 4.5 CASS outage

Required fallback behavior:

- post-mortem drafts go to `docs/sessions/` locally,
- telemetry persists to `.pi-flywheel/error-counts.json` plus any session spool,
- any UI or report that would have shown CASS-backed insights should say
  "stored locally; CASS unavailable".

No subsystem in this bundle should hard-fail purely because `cm` is absent.

### 4.6 Checkpoint corruption

Required fallback behavior:

- checkpoint parse or hash failure is not treated as a total session-loss event,
- reconstruct from git log, bead IDs, and any in-memory results still present,
- quarantine bad checkpoint content using the existing `.corrupt` flow,
- never rewrite a guessed checkpoint in place during post-mortem generation.

### 4.7 Agent Mail contact policy edge

Prior context said first contact to the coordinator could fail due to
`contacts_only`.

Robust plan:

- add a coordinator bootstrap bead that sets `contact_policy=auto` as part of
  role-aware startup,
- keep it idempotent,
- and treat failure to set policy as a warning in doctor/bootstrap, not an
  invisible footgun discovered at handoff time.

### 4.8 Interaction with in-process mutexes

Current `mutex.ts` only protects concurrent calls inside one process.

Implication:

- do not rely on it for post-mortem or telemetry file safety across processes,
- do not present its protection as if it were global,
- and prefer session-scoped files plus atomic rename over shared mutable blobs.

## 5. Resource bounds

### 5.1 Memory bounds for error-code telemetry

Target session:

- duration: 4 hours,
- total emitted structured error events: 10,000+,
- expected distinct codes after v3.4.0 additions: still comfortably under 32.

Bounded in-memory design:

- primary structure: `Map<FlywheelErrorCode, CounterState>`,
- one counter object per code,
- `CounterState` includes:
  - `count`,
  - `firstSeenAt`,
  - `lastSeenAt`,
  - `lastContext` with trimmed tool/phase/bead/timestamp,
  - `sampleRing` with max 8 sanitized samples per code.

Global caps:

- `maxCodesTracked = 64` hard cap,
- `maxSamplesPerCode = 8`,
- `maxSampleTextBytes = 256`,
- `globalRecentRing = 128` for cross-code recent samples,
- `maxEstimatedMemory = 256 KB` soft cap.

If the soft cap estimate is exceeded:

- stop recording new samples,
- keep only counts,
- increment `droppedSamples`,
- emit one warning at flush time.

### 5.2 Disk bounds for post-mortem drafts

Path policy:

- local drafts live under `docs/sessions/`,
- filename format should include date and session ID, e.g.
  `docs/sessions/2026-04-21-<sessionId>.md`.

Overwrite policy:

- if the same session ID regenerates the draft, overwrite only when the file
  contains the autogen marker and the user requested regeneration,
- otherwise write `-r2`, `-r3`, and so on.

Quota policy:

- soft cap: 100 autogen session files,
- soft size cap: 10 MB total autogen markdown under `docs/sessions/`,
- hard failure is undesirable; warn once above soft cap and keep writing.

Why warn instead of auto-delete:

- silent deletion of retrospectives is a worse failure than local repo bloat,
- and cleanup is a maintenance task that deserves an explicit bead or command.

### 5.3 Disk bounds for telemetry fallback

Fallback path:

- `.pi-flywheel/error-counts.json` remains the compatibility summary file.

Recommended companion path:

- `.pi-flywheel/error-counts/session-<sessionId>.json` for spool data.

Bounds:

- summary file should keep only the most recent 20 sessions,
- per-session file should be capped to counts plus bounded samples only,
- total summary target: < 256 KB,
- total per-session spool target: < 32 KB per session.

### 5.4 Per-check timeouts for doctor

Target:

- full green sweep should return in under 5 seconds wall-clock.
- This requires parallel execution with per-check timeouts.

Concrete timeout table:

| Check | Command | Timeout |
|---|---|---:|
| repo-root sanity | `git rev-parse --is-inside-work-tree` | 150 ms |
| `git` presence | `git --version` | 150 ms |
| `node` presence | `node --version` | 150 ms |
| `br` presence | `br --version` | 250 ms |
| `bv` presence | `bv --help` | 300 ms |
| `ntm` presence | `ntm --help` | 350 ms |
| `cm` presence | `cm --version` | 300 ms |
| Agent Mail liveness | `curl -s -I http://127.0.0.1:8765/mcp` | 500 ms |
| dist heuristic | fast `git diff --name-only` heuristic | 350 ms |
| checkpoint read | no subprocess; in-process parse | 50 ms budget |

Parallelization plan:

- batch 1: repo, git, node, br, bv, ntm, cm, dist heuristic in parallel,
- batch 2: Agent Mail liveness in parallel with batch 1,
- checkpoint read in-process concurrently.

Worst-case wall clock under green conditions:

- dominated by the 500 ms Agent Mail check plus small overhead,
- still under 1 second on a healthy machine,
- with ample room below the 5 second goal.

Worst-case wall clock under degraded conditions:

- still bounded by the slowest timeout rather than a serial sum,
- because the implementation uses `Promise.allSettled`.

### 5.5 Subprocess concurrency limits

Doctor should not spawn an unbounded number of child processes.

Limit:

- max 6 concurrent child processes at once.

Reasoning:

- the doctor only needs a handful of short-lived checks,
- the rest of the bundle’s heavy operations should avoid fan-out,
- and keeping the cap low reduces contention with the user’s own shell or other
  running sessions.

### 5.6 Hotspot analysis bounds

Hotspot matrix limits:

- max detail rows emitted: 25,
- max raw artifact mentions processed before summarizing: 2,000,
- max beads per standalone run before summary mode: 150.

If bounds are exceeded:

- compute counts,
- emit summary statistics,
- and include only top 25 contested files.

### 5.7 Post-mortem bounds

Draft bounds:

- max commit lines listed: 20,
- max bead IDs listed in full: 30,
- max Agent Mail subject lines: 10,
- max CASS bullets summarized: 10,
- max draft size target: 12 KB markdown before attachments or extras.

If the draft would exceed target:

- collapse sections to summary sentences,
- keep the warning section,
- and prioritize recent failures over exhaustive histories.

## 6. Cancellation and signals

### 6.1 Required rule

Every new subprocess introduced by this bundle must:

- use `ExecFn`,
- include a concrete `timeout`,
- and pass `signal` through unchanged when a caller received one.

This is not optional. It is already a repo rule and should be treated as a
review-blocking defect if violated.

### 6.2 `cli-exec.ts` must stop dropping `signal`

Current state:

- `mcp-server/src/cli-exec.ts` forwards `cwd` and `timeout`,
- but not `signal`.

Required change:

- extend `ResilientExecOptions` with `signal?: AbortSignal`,
- pass `signal` through into the underlying `exec`,
- and ensure retry loops bail immediately if `signal.aborted` becomes true.

Without this, any new doctor/post-mortem/telemetry code using the resilient
wrappers will quietly violate the repo’s cancellation contract.

### 6.3 New exec call-site map

The list below names every planned new subprocess call site and the required
signal-threading behavior.

| File / function | Command(s) | Signal required? | Notes |
|---|---|---|---|
| `mcp-server/src/tools/doctor.ts::runDoctor` | `git`, `node`, `br`, `bv`, `ntm`, `cm`, `curl` | yes | every row is cancellable |
| `mcp-server/src/tools/doctor.ts::checkAgentMailHealth` | `curl` | yes | short timeout, no retries after abort |
| `mcp-server/src/episodic-memory.ts::generatePostmortemDraft` | `git` | yes | commit summary and HEAD checks |
| `mcp-server/src/episodic-memory.ts::collectBeadSummary` | `br` | yes | fallback to checkpoint on failure |
| `mcp-server/src/episodic-memory.ts::storePostmortemToCass` | `cm add` | yes | skip on abort |
| `mcp-server/src/telemetry.ts::flushTelemetryToCass` | `cm add` | yes | re-entrancy guarded |
| `mcp-server/src/telemetry.ts::loadGitContextForSummary` | `git log` | yes | optional severity summary |
| `mcp-server/src/agent-mail.ts::checkAgentMailHealth` if added | `curl` | yes | aligns with AGENTS guidance |

Planned no-exec additions:

- `plan-simulation.ts` hotspot matrix should stay pure.
- `bead-templates.ts` validation and versioning should stay pure.
- `deep-plan.ts` template metadata plumbing should stay file/string based, not
  subprocess based.

### 6.4 Mid-doctor Ctrl-C behavior

Required behavior:

- settled rows remain visible,
- in-flight rows become `cancelled`,
- return path is a structured `exec_aborted` error with a partial report,
- and no lingering child processes survive.

Why partial rows matter:

- users otherwise cannot tell whether doctor failed early or proved some
  dependencies healthy before cancellation,
- and repeated retries become slower if they must rediscover all context.

### 6.5 Post-mortem and telemetry abort behavior

Post-mortem:

- if cancelled before file write starts, write nothing,
- if cancelled after temp file write but before rename, remove temp if possible,
- if cancelled after rename, treat the draft as successfully emitted and do not
  try to roll it back.

Telemetry:

- on abort, skip CASS store,
- attempt only the fastest local fallback if it can finish immediately,
- never let telemetry flush block an abort from reaching the user.

### 6.6 Tests required for signal handling

- doctor row timeout and abort tests,
- resilient exec retry cancellation tests,
- post-mortem abort leaves no temp file,
- telemetry abort does not recursively emit more telemetry,
- any added Agent Mail health helper respects abort.

## 7. New `FlywheelErrorCode` entries

The bundle can lean on existing generic codes (`exec_timeout`, `exec_aborted`,
`cli_failure`, `parse_failure`) for many branches. New codes should exist only
where the orchestrator needs a distinct branch or a user-facing explanation
that generic codes would obscure.

### 7.1 Proposed additions

| Code | Retryable | When thrown | Orchestrator behavior |
|---|---|---|---|
| `doctor_blocking_failure` | false | direct `flywheel_doctor` call finds required dependency failure | ask user to fix or continue degraded; do not auto-launch dependent workflows |
| `doctor_partial_result` | true | doctor was cancelled after producing partial rows | show partial report and offer retry |
| `hotspot_input_unreliable` | false | standalone hotspot request lacks trustworthy file-scope inputs | fall back to conservative launch advice; no hard abort of approval |
| `postmortem_checkpoint_invalid` | false | checkpoint missing/corrupt for a direct post-mortem generation call | reconstruct from git/beads or emit local-only draft |
| `postmortem_reconstruction_failed` | true | both checkpoint and reconstruction inputs are insufficient | emit minimal stub draft and recommend manual review |
| `template_expansion_failed` | false | template lookup or placeholder expansion fails | block only the affected template path; require correction or custom fallback |
| `template_version_conflict` | false | plan pins template version that no longer matches loaded library | ask for regen or use pinned stored copy |
| `telemetry_store_failed` | true | CASS flush fails and local fallback is used | continue shutdown and note local persistence |
| `telemetry_merge_conflict` | true | shared fallback summary cannot be merged after retries | keep session spool file and continue |

### 7.2 Why these need dedicated codes

`doctor_blocking_failure`

- Generic `cli_failure` does not say whether the session may continue.
- The orchestrator needs to present "fix vs continue degraded" rather than just
  dumping stderr.

`doctor_partial_result`

- `exec_aborted` alone does not imply that partial rows exist.
- The skill and UI can do more with a code that explicitly says a partial report
  is available.

`hotspot_input_unreliable`

- This is not a generic parse failure.
- It describes a specific advisory failure: the data exists but is not good
  enough to drive strong contention conclusions.

`postmortem_checkpoint_invalid`

- This tells the Step 10 wrap-up path to switch reconstruction mode rather than
  treating the whole feature as dead.

`postmortem_reconstruction_failed`

- This captures the case where both primary and fallback sources are weak, which
  deserves a minimal draft rather than silent disappearance.

`template_expansion_failed`

- A template failure should branch differently than a generic invalid input.
- The planner or bead-creation flow can retry with a custom description.

`template_version_conflict`

- Version drift is a domain-specific migration problem, not a generic parse
  problem.

`telemetry_store_failed`

- The orchestrator must know that telemetry still exists locally and should not
  tell the user it was lost.

`telemetry_merge_conflict`

- This distinguishes "CASS failed" from "local merge failed but data is still
  recoverable from spool files".

### 7.3 SKILL.md / orchestrator reactions

- `doctor_blocking_failure`
  Step 0 or direct doctor command asks:
  fix now, continue degraded, or show remediation details.
- `doctor_partial_result`
  Present partial rows and a retry option.
- `hotspot_input_unreliable`
  Approval falls back to safer launch advice and shows lower confidence.
- `postmortem_checkpoint_invalid`
  Wrap-up uses reconstruction mode and warns in the generated artifact.
- `postmortem_reconstruction_failed`
  Wrap-up emits a minimal local draft and suggests manual completion.
- `template_expansion_failed`
  Planner/bead generator retries with custom prose rather than raw template
  markers.
- `template_version_conflict`
  Plan approval asks for regeneration or template pinning resolution.
- `telemetry_store_failed`
  Step 10 summary says telemetry stored locally only.
- `telemetry_merge_conflict`
  Step 10 summary says local spool preserved; compaction deferred.

## 8. Plan-simulation determinism

### 8.1 Deterministic input normalization

Required rules:

- sort beads by `id` before any hotspot iteration,
- sort dependency arrays before computing waves,
- normalize and sort artifact paths per bead,
- dedupe paths after normalization,
- sort final rows by:
  1. severity descending,
  2. write count descending,
  3. canonical file path ascending.

### 8.2 Confidence tiers for extracted paths

To address the false-positive discussion requested in the task:

| Source of path mention | Confidence | Used for high severity? |
|---|---|---|
| `### Files:` section | high | yes |
| bullet under files section | high | yes |
| root-level bullet path outside files section | medium | yes, with warning |
| free prose mention | low | no, advisory only |

Rule:

- low-confidence matches may create a warning row,
- but cannot alone escalate a file to `high` severity.

This biases toward precision rather than recall for launch gating, which is the
right tradeoff here because false-positive serialization hurts throughput while
false-negative hazards can still be caught by file reservations at runtime.

### 8.3 Precision vs recall tradeoff

Precision-first advantages:

- fewer false "do not swarm" results,
- fewer debates about clearly non-writing references,
- more trust in high-severity rows.

Recall-first advantages:

- catches more possible hazards,
- useful for manual reviewer attention.

Chosen policy:

- use precision-first for the launch recommendation,
- use recall-friendly warnings for the human-readable appendix,
- and always show why a path was counted.

### 8.4 Critical-file boost

Determinism is not just ordering. It is also semantic stability.

If `server.ts`, `types.ts`, `errors.ts`, `checkpoint.ts`, or `agent-mail.ts`
are contested:

- severity should deterministically lift to `high`,
- regardless of raw count,
- because these are contract-heavy shared surfaces.

### 8.5 Snapshot semantics

Hotspot analysis should state explicitly:

- the matrix is based on a bead snapshot taken at a specific time,
- later edits require rerun,
- and the matrix is not a live lock manager.

This wording is part of robustness because it prevents the tool from being
misunderstood as stronger than it is.

## 9. Upgrade / migration robustness

### 9.1 Telemetry schema versioning

Every telemetry payload stored to CASS or local fallback should include:

- `schemaVersion`,
- `producerVersion` (flywheel version),
- `sessionId`,
- `startedAt`,
- `endedAt`.

Reader policy:

- accept known current schema,
- tolerate newer schema by reading common fields only,
- ignore unknown fields,
- never fail the whole telemetry summary because one historical entry is newer.

### 9.2 Mixed-version CASS entries

Required behavior:

- a v3.5 writer must not break v3.4 readers,
- a v3.4 reader should surface "newer schema ignored partially" rather than
  throwing,
- severity/count summaries must use common stable keys only.

### 9.3 Bead template versioning

Template objects should gain explicit version identity.

Recommended shape:

- template ID remains stable, e.g. `spawn-swarm`,
- template version is additive, e.g. `1`,
- plans may reference `spawn-swarm@1`.

If a plan references a template with no explicit version:

- treat it as "latest at synthesis time",
- warn on later expansion if the current version differs and no pinned copy is
  available,
- prefer explicit pinning for saved plan artifacts.

### 9.4 Existing plan compatibility

Older plans and prompts that know nothing about template versions should still
work.

Compatibility rule:

- if no version is present, expansion is allowed,
- but the expansion result should include a warning when the template library
  has materially changed since the plan was written.

### 9.5 Checkpoint schema stability

Do not force a checkpoint schema bump for this bundle unless absolutely needed.

Preferred approach:

- keep new post-mortem and telemetry state ephemeral or separately persisted,
- avoid adding required checkpoint fields,
- derive session summaries from existing `FlywheelState` plus git data.

This minimizes recovery risk and avoids accidental breakage of the current
`schemaVersion: 1` checkpoint reader.

### 9.6 Feature flags and rollback safety

Recommended kill switches:

- `FLYWHEEL_DOCTOR_ENABLED=0`
- `FLYWHEEL_HOTSPOT_ENABLED=0`
- `FLYWHEEL_POSTMORTEM_ENABLED=0`
- `FLYWHEEL_TEMPLATE_VERSION_ENFORCED=0`
- `FLYWHEEL_TELEMETRY_ENABLED=0`

Reason:

- observability features should be easy to disable if they regress behavior,
- especially at release time,
- without reverting the entire bead-quality or review pipeline.

## 12. Verification block

The verification plan must include chaos, boundedness, concurrency, and
determinism checks. This is not optional for a release that claims robustness.

### 12.1 Chaos-style tests

1. Kill `br` mid-doctor
   - Method: mocked exec or subprocess kill during `br --version`
   - Expectation: doctor row goes red/yellow with timeout/abort details; tool
     returns remaining rows and never hangs

2. Kill `curl` during Agent Mail liveness check
   - Method: abort or kill the `curl` child
   - Expectation: doctor reports Agent Mail degraded and does not claim swarm
     readiness

3. Kill `cm` during post-mortem CASS store
   - Method: mocked `cm add` aborts or exits non-zero
   - Expectation: local draft still exists; CASS store warning recorded

4. Kill `cm` during telemetry flush
   - Method: abort while flushing
   - Expectation: telemetry falls back locally and does not recurse into more
     telemetry errors

5. Kill doctor with Ctrl-C after partial settle
   - Method: abort signal after some checks resolve
   - Expectation: partial report includes cancelled rows

### 12.2 Bounded tests

1. Run doctor 100 times in a loop
   - Expectation: no unbounded memory growth and stable row ordering

2. Feed telemetry 10,000 synthetic error events
   - Expectation: counters are accurate, dropped-sample count increments after
     the bound, and heap growth remains under target

3. Generate post-mortem from oversized CASS input
   - Expectation: output remains within draft-size bound and includes truncation
     warning instead of dumping raw payload

4. Feed hotspot analysis 150 beads with 2,000 path mentions
   - Expectation: execution time remains bounded; output truncates to top rows

### 12.3 Concurrency tests

1. Two sessions run doctor on the same repo
   - Expectation: both complete, both report contention warning, neither locks
     the other out

2. Two sessions crash and flush telemetry
   - Expectation: both session spool records survive; summary merge is valid or
     explicitly recoverable

3. Two sessions emit post-mortems on the same date
   - Expectation: filenames differ by session ID or revision suffix, with no
     overwrite

4. Approval runs hotspot analysis while another process mutates beads
   - Expectation: analysis is labeled snapshot-based and recommends rerun rather
     than mixing stale and fresh state invisibly

### 12.4 Determinism tests

1. Reverse bead input order for hotspot analysis
   - Expectation: identical JSON and markdown output

2. Normalize equivalent paths (`./src/foo.ts`, `src/foo.ts`)
   - Expectation: one canonical row

3. Rerun same plan with same inputs across two processes
   - Expectation: same matrix, same severity ordering, same recommendation

### 12.5 Regression tests from prior-session context

1. Missing Gemini / `ntm --gmi` unavailable
   - Expectation: doctor warns, does not crash

2. Coordinator `contacts_only`
   - Expectation: coordinator bootstrap auto-sets `contact_policy=auto`

3. Already-closed bead review parse
   - Expectation: `flywheel_review` still reconciles closed beads without parse
     failure

4. Spawn-swarm template pane semantics
   - Expectation: examples and generated text use integer `--pane`

5. Monitor tool noisy exit
   - Expectation: template-generated informational monitors use `|| true`

6. Corrupt checkpoint
   - Expectation: post-mortem reconstructs and warns

7. Huge CASS payload
   - Expectation: post-mortem and telemetry summarize rather than dump

### 12.6 Release exit criteria

Do not merge the bundle until all of the following are true:

- doctor direct-call tests pass,
- hotspot determinism tests pass,
- post-mortem corrupt-checkpoint fallback tests pass,
- template version-conflict tests pass,
- telemetry 10k-event boundedness tests pass,
- concurrent crash merge tests pass,
- three fresh-eyes gates are closed,
- and `dist/` is rebuilt with no drift.

## 13. Recommended rollout order

1. Land the foundational error/signal plumbing first.
2. Ship doctor and hotspot next because they improve observability for the
   remaining work.
3. Land post-mortem and template hardening together because the template wave
   also needs to encode the `ntm --pane=<int>` and monitor `|| true`
   regressions.
4. Land telemetry last because it is the easiest feature to make recursive or
   unbounded if the earlier contracts are not already stable.
5. Use fresh-eyes review gates after each wave, not just at the end.

## 14. Final recommendation

This bundle is worth shipping in v3.4.0 only if the implementation stays
strictly additive and bounded.

The main failure patterns to defend against are:

- false confidence from heuristics,
- cancellation leaks from wrappers that drop `signal`,
- silent local-data corruption when two sessions write at once,
- and "optional dependency missing" paths that accidentally abort whole flows.

If the team needs to cut scope, the safest order to defer is:

1. fancy template-version features,
2. richer post-mortem highlights,
3. telemetry severity-side summaries,

while keeping:

- doctor,
- hotspot determinism,
- local fallback post-mortems,
- and bounded telemetry counters.

Those are the highest-value robustness wins for the release.
