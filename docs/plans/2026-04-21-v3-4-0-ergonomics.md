## 0. Executive Summary

v3.4.0 bundles five accretive observability and reliability wins. This plan focuses on the ergonomic layer: invocation paths, output shape, AskUserQuestion integration, failure UX, default vs override, and the exact SKILL.md touch points every implementation agent must hit.

The north star: a user who has never read the changelog should stumble into all five features naturally during a normal flywheel session and find them self-explanatory. A future AI agent reading SKILL.md cold should know exactly when to call each feature and what to do with the result.

## 1. Subsystem UX Details

### 1.1 `flywheel_doctor` — Dependency & Health Diagnostic

#### Invocation
Two entry points:

**A) Slash command (user-initiated):**
```
/flywheel-doctor
```
Runs immediately with all checks; no arguments required. The command `.md` accepts one optional flag:
```
/flywheel-doctor --checks=mcp,am,git   # comma-separated subset
/flywheel-doctor --format=json         # machine-readable output for CI
```

**B) Implicit — Step 0b of SKILL.md (`skills/start/SKILL.md`):**
After completing checks 1–7 in the existing Step 0b block, the orchestrator calls `flywheel_doctor` silently as check 8. Results are folded into the banner. No additional user input required unless a red check is found.

**Concrete MCP tool call (CI usage):**
```ts
flywheel_doctor({ cwd: "/path/to/repo", checks: ["all"], format: "structured" })
```

#### Output Shape

Terminal output uses a consistent three-symbol scheme throughout v3.4.0:
- `[OK]` — green; check passed
- `[WARN]` — yellow; check passed with caveats; session can continue
- `[FAIL]` — red; check failed; action required or recommended

```
══════════════════════════════════════════
  flywheel doctor  v3.4.0
══════════════════════════════════════════
[OK]   MCP connectivity         flywheel_profile responded in 42ms
[OK]   Agent Mail liveness      http://127.0.0.1:8765 — healthy
[OK]   br (bead runner)         v2.1.0 at /usr/local/bin/br
[WARN] bv (bead verifier)       not found — verify-beads step will degrade
       hint: brew install bead-runner  OR  npx @pitools/bv
[OK]   ntm                      v1.3.2; projects_base=/Users/kev/projects [matched]
[OK]   cm (CASS)                v0.9.1 — memory store available
[OK]   node                     v22.3.0 (>=20 required)
[OK]   git                      v2.44.0
[OK]   dist-drift               mcp-server/dist/ matches src/ (last built 3m ago)
[WARN] orphaned worktrees       2 worktrees >7 days old with no open beads
       hint: run /flywheel-cleanup to remove them
[OK]   checkpoint.json          phase=complete — no stale session
══════════════════════════════════════════
  3 checks total: 8 OK, 2 WARN, 0 FAIL
  Session ready. WARN items are non-blocking.
══════════════════════════════════════════
```

If any `[FAIL]` is present:
```
══════════════════════════════════════════
  flywheel doctor  v3.4.0
══════════════════════════════════════════
[OK]   MCP connectivity         flywheel_profile responded in 38ms
[FAIL] Agent Mail liveness      connection refused — http://127.0.0.1:8765
       code: agent_mail_unreachable
       hint: run `agent-mail start` or check if the server process exited
[OK]   br (bead runner)         v2.1.0
...
══════════════════════════════════════════
  1 FAIL — parallel agent spawning will be unavailable this session
══════════════════════════════════════════
```

Structured JSON output (for CI / `--format=json`):
```json
{
  "version": 1,
  "status": "warn",
  "checks": [
    { "name": "mcp",      "status": "ok",   "detail": "flywheel_profile OK in 42ms" },
    { "name": "am",       "status": "warn", "detail": "...", "code": "agent_mail_unreachable", "hint": "..." },
    { "name": "bv",       "status": "warn", "detail": "not found", "hint": "brew install bead-runner" }
  ],
  "summary": { "ok": 8, "warn": 2, "fail": 0 }
}
```

#### AskUserQuestion Integration

Only triggered on `[FAIL]` findings during Step 0b. Presented at most once per session, batching all failures into a single question:

```
AskUserQuestion(questions: [{
  question: "flywheel doctor found 1 FAIL: Agent Mail unreachable. How should I proceed?",
  header: "Doctor: Action Required",
  options: [
    { label: "Fix and retry", description: "I'll start Agent Mail — re-run doctor check after" },
    { label: "Continue single-agent", description: "Skip parallel agents this session (safe for small goals)" },
    { label: "View remediation", description: "Show full hint and error details before deciding" }
  ],
  multiSelect: false
}])
```

On `[WARN]` only: no question. Warnings are displayed in the banner and silently noted in the session log. The user can always call `/flywheel-doctor` again.

#### Failure UX

All failures from `flywheel_doctor` itself follow the v3.3.0 error contract:

| Scenario | Code | hint |
|---|---|---|
| Tool call times out (>8s) | `exec_timeout` | "Doctor timed out — check MCP server process; retry with --checks=subset" |
| Individual check parse failure | `parse_failure` | "Check `<name>` output could not be parsed — treating as WARN; see stderr for raw output" |
| Missing cwd | `missing_prerequisite` | "cwd must point to a git repo root with .pi-flywheel/" |
| dist-drift check fails to build | `cli_failure` | "npm run build exited non-zero — run `cd mcp-server && npm run build` and check errors" |

Doctor itself never throws on check failures — only on its own invocation failures. Individual check failures are downgraded to `[WARN]` with a structured hint.

#### Default vs Override

| Behavior | Default | Override |
|---|---|---|
| Checks run | all 10 | `--checks=mcp,git,am` per-invocation |
| Trigger in Step 0b | always (silent) | `FLYWHEEL_DOCTOR_SKIP=1` env var disables |
| Orphaned-worktree age threshold | 7 days | `FLYWHEEL_ORPHAN_DAYS=N` env var |
| dist-drift timeout | 30s | `FLYWHEEL_DIST_TIMEOUT_MS=N` env var |
| Output format | human-readable | `--format=json` per-invocation; `FLYWHEEL_DOCTOR_FORMAT=json` persistent |

### 1.2 Shared-Write Hotspot Report

#### Invocation

**A) Implicit — Step 6 of `_beads.md`:**
`flywheel_approve_beads` automatically runs the hotspot matrix when beads are loaded. The matrix appears in the approve-beads output above the quality score. No user action required to see it.

**B) Standalone MCP call:**
```ts
flywheel_approve_beads({ cwd, action: "hotspot" })
```
Returns only the contention matrix without triggering approval flow.

**C) Via `plan-simulation.ts` directly (CI/testing):**
```ts
import { buildContentionMatrix } from './plan-simulation.js';
const matrix = buildContentionMatrix(beads);
```

#### Output Shape

Surfaced in the Step 6 approve-beads output, between the bead list and the quality score:

```
── Shared-Write Hotspot Report ──────────────────────────────
File                                  Beads        Risk
mcp-server/src/plan-simulation.ts     B-3, B-5     [HIGH]  ← 2 parallel beads write this file
mcp-server/src/server.ts              B-1, B-4     [MED]
skills/start/_beads.md                B-2, B-6     [LOW]   ← different sections
─────────────────────────────────────────────────────────────
Recommendation: run B-3 and B-5 COORDINATOR-SERIAL (one agent at a time)
                B-1 and B-4 can run swarm with file-reservation guard

[OK]  No critical contention — swarm mode safe with reservations
```

Risk levels:
- `[HIGH]` — 3+ beads write same file in same wave, or 2 beads with no dependency edge writing a critical shared file (server.ts, types.ts, errors.ts)
- `[MED]` — 2 beads write same file; addressable with file reservation
- `[LOW]` — 2 beads write same file but in non-overlapping sections (heuristic: different top-level functions detected)

Launch mode recommendation auto-populates the Step 6 launch-gate question:

```
AskUserQuestion(questions: [{
  question: "Quality score: 0.87/1.00. Hotspot analysis recommends coordinator-serial for B-3/B-5 (plan-simulation.ts conflict). Launch mode?",
  header: "Launch",
  options: [
    { label: "Swarm (recommended base)", description: "Agents run in parallel with file-reservation guards" },
    { label: "Coordinator-serial for hotspots", description: "B-3 and B-5 run sequentially; others parallel (Recommended by hotspot analysis)" },
    { label: "Full coordinator-serial", description: "All beads run one at a time — slowest but safest" },
    { label: "Polish more", description: "Restructure beads to reduce contention before launching" }
  ],
  multiSelect: false
}])
```

This replaces the current two-option launch menu when hotspot `[HIGH]` or `[MED]` findings are present.

#### Failure UX

| Scenario | Code | hint |
|---|---|---|
| Bead body parse fails (no file mentions found) | `parse_failure` | "Hotspot analysis found no file paths in bead bodies — ensure beads follow body discipline. Treating all as LOW contention." |
| No beads loaded | `empty_plan` | "No beads found — run bead creation first" |
| Matrix build timeout (>5s) | `exec_timeout` | "Hotspot analysis timed out — proceeding without matrix; treat all beads as independent" |

On parse failure the hotspot section is omitted from output; launch gate proceeds with the current two-option menu. The session log records `[WARN] hotspot matrix skipped: parse_failure`.

#### Default vs Override

| Behavior | Default | Override |
|---|---|---|
| Run on `flywheel_approve_beads` | always | `FLYWHEEL_HOTSPOT_SKIP=1` disables |
| HIGH threshold (bead count) | 2+ beads, critical file | configurable in server settings |
| Recommendation injection into AskUserQuestion | always when HIGH/MED | `--no-hotspot-hint` per-invocation |

### 1.3 Post-Mortem CASS Auto-Generation

#### Invocation

**A) Implicit — Step 10 of `_wrapup.md`:**
After `flywheel_memory(operation: "store")`, the orchestrator auto-drafts a post-mortem. The draft is presented for user review before committing.

**B) On `/flywheel-stop`:**
If the user ends a session mid-flight, `flywheel_stop` checks for `checkpoint.json` and auto-generates a partial post-mortem with whatever data is available.

**C) Manual:**
```ts
flywheel_memory({ cwd, operation: "draft_postmortem" })
```

#### Output Shape

Step 10 presents the draft inline in the terminal before asking the user to confirm or edit:

```
── Post-Mortem Draft ────────────────────────────────────────
Session: 2026-04-21  Goal: "Ship v3.3.0 error contracts"
Commits: 4 (a9fcd85..5d5fd94)

## What worked well
- AskUserQuestion batching: alignment converged in 1 cycle vs 3
- Per-bead mutex prevented 2 concurrent_write collisions

## What required intervention
- Agent Mail was unreachable at session start (agent_mail_unreachable x1)
  → Single-agent fallback worked; no lost work

## Error codes this session
- concurrent_write: 2  (both auto-resolved by mutex)
- agent_mail_unreachable: 1

## Decisions made
- Chose coordinator-serial for B-3/B-5 due to plan-simulation.ts contention
- Deferred UI polish pass to v3.4.0

## Patterns to replicate
- Batch alignment questions 3-at-a-time

─────────────────────────────────────────────────────────────
```

Then:
```
AskUserQuestion(questions: [{
  question: "Post-mortem draft ready. What would you like to do?",
  header: "Post-mortem",
  options: [
    { label: "Commit as-is", description: "Store this draft to CASS as session learnings (Recommended)" },
    { label: "Edit first", description: "I'll edit the draft at .pi-flywheel/postmortem-draft.md then continue" },
    { label: "Discard", description: "Skip post-mortem — no learnings stored this session" }
  ],
  multiSelect: false
}])
```

- "Commit as-is" → call `flywheel_memory(operation: "store")` with draft content
- "Edit first" → write draft to `.pi-flywheel/postmortem-draft.md`; pause and wait for user; when they return, offer to re-read and store
- "Discard" → skip; log `post_mortem_skipped` in session summary

#### Failure UX

| Scenario | Code | hint |
|---|---|---|
| `git log` fails (no commits in range) | `cli_failure` | "No commits found since session start — post-mortem will omit commit section" |
| checkpoint.json missing or empty | `not_found` | "No checkpoint found — post-mortem drafted from Agent Mail inbox only" |
| CASS store fails | `cli_failure` | "CASS store failed — draft saved to .pi-flywheel/postmortem-draft.md for manual import" |
| Draft is empty (no data at all) | `empty_plan` | "Not enough session data to draft a post-mortem — session was too short or had no tool calls" |

On empty draft: skip the review step entirely; log `post_mortem_empty` in Step 10 summary. Do not show an empty review to the user.

#### Default vs Override

| Behavior | Default | Override |
|---|---|---|
| Auto-draft on Step 10 | always | `FLYWHEEL_POSTMORTEM_SKIP=1` disables |
| Auto-draft on `/flywheel-stop` | always | same env var |
| Draft file location | `.pi-flywheel/postmortem-draft.md` | not configurable (always in session dir) |
| Minimum commits to include commit section | 1 | `FLYWHEEL_POSTMORTEM_MIN_COMMITS=0` includes even zero-commit sessions |

### 1.5 Error-Code Telemetry

#### Invocation

**A) Automatic — session end (Step 10 / `/flywheel-stop`):**
Aggregation runs silently. No user action needed.

**B) Surfaced in Step 10 summary banner:**
After the welcome banner in a new session (Step 0c), if 10+ prior sessions exist in CASS, the orchestrator prepends:
```
Error-code trends (last 10 sessions):
  concurrent_write      8 occurrences  [trending UP ↑]
  agent_mail_unreachable  3 occurrences
  parse_failure           1 occurrence
```

**C) Explicit query:**
```ts
flywheel_memory({ cwd, operation: "search", query: "error_code_telemetry top_n" })
```

#### Output Shape

In the Step 0c banner (prepended before the main menu):
```
┌─ Error Trends · last 10 sessions ──────────────────────────┐
│  concurrent_write        8  ↑  (was 3 in prior 10)         │
│  agent_mail_unreachable  3                                  │
│  parse_failure           1                                  │
│  (7 other codes: 0 occurrences)                             │
└─────────────────────────────────────────────────────────────┘
```
Only shown if at least one non-zero code exists. Hidden if all codes are 0 (clean bill of health).

In Step 10 post-mortem draft (auto-included):
```
## Error codes this session
- concurrent_write: 2  (mutex auto-resolved)
- agent_mail_unreachable: 1  (single-agent fallback)
```

#### AskUserQuestion Integration

No AskUserQuestion for telemetry collection — it's fully automatic. The only user-facing moment is the trend banner, which is read-only. If `concurrent_write` is trending up sharply (>2x), a soft warning is inserted:

```
[WARN] concurrent_write trending up — consider adding file-reservation guards
       or increasing coordinator-serial usage in high-contention beads.
       Run /flywheel-doctor for details.
```

#### Failure UX

| Scenario | Code | hint |
|---|---|---|
| CASS write fails at session end | `cli_failure` | "Telemetry could not be persisted — session data not lost, but trends may be incomplete" |
| CASS read fails at session start | `cli_failure` | "Could not load error trends — proceeding without trend banner" |
| checkpoint.json lacks `errorCodes` field (old format) | graceful degradation | No error shown; field treated as empty map `{}` |

Telemetry NEVER blocks session start or end. Any failure in the telemetry path logs a warning to stderr and continues.

#### Default vs Override

| Behavior | Default | Override |
|---|---|---|
| Collect at session end | always | `FLYWHEEL_TELEMETRY_SKIP=1` disables |
| Show trend banner | only if ≥1 non-zero code | `FLYWHEEL_TELEMETRY_ALWAYS_SHOW=1` shows even when all zero |
| Trend lookback window | 10 sessions | `FLYWHEEL_TELEMETRY_WINDOW=N` |
| "trending up" threshold | 2x increase | not configurable in v3.4.0 |

doctor-smoke-test:

## 6. Migration and Backward Compatibility

### checkpoint.json (old format)

Fields added in v3.4.0: `errorCodeTelemetry: Record<FlywheelErrorCode, number>`.

**Graceful degradation**: `loadState()` in `state.ts` already returns `createInitialState()` for fields not present in disk state. The new `errorCodeTelemetry` field defaults to `{}` (empty map). No migration script needed. Old checkpoints load cleanly; telemetry simply starts from zero for the first v3.4.0 session.

### CASS entries (old sessions)

Post-mortem draft format is new. Old CASS entries lack the `## Error codes this session` section. The telemetry aggregator reads only entries that contain `error_code_telemetry` in their metadata. Missing metadata is treated as `{}`. Trend computation over 10 sessions correctly handles mixed old/new entries (old = all zeros for that session).

### bead-templates.ts (existing templates)

## 9. Open Questions for Coordinator / Other Planners

1. **doctor timeout budget**: Is 8s per check acceptable or should individual checks have per-check timeouts (e.g., 3s for AM, 30s for dist-drift)? Ergonomics preference: per-check timeouts with a 60s total cap.

2. **Hotspot LOW threshold**: The "different sections" heuristic (§1.2) requires parsing function boundaries. Should v3.4.0 ship with a simpler line-number heuristic (top/bottom 25% of file) instead? Ergonomics preference: simpler heuristic — accuracy over false precision.

3. **Post-mortem "Edit first" UX**: When user selects "Edit first", should the orchestrator actively watch the draft file for changes (polling) or simply wait for the user to say "done"? Ergonomics preference: wait for user message ("I'm done editing") — polling is fragile.

4. **Trend window persistence**: Should `FLYWHEEL_TELEMETRY_WINDOW` be a per-project setting (stored in `.pi-flywheel/settings.json`) or env-var only? Ergonomics preference: env-var only in v3.4.0; per-project settings deferred to v3.5.

5. **Template count cap**: `AskUserQuestion` max 4 options. With 11 templates, "Browse templates" needs pagination. Should v3.4.0 cap at 4 most-relevant templates (filtered by plan scan) or implement two-page browsing? Ergonomics preference: filter to 4 most-relevant; show full list only if user selects "See all".
